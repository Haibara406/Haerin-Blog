# A技术点总结

## damai-common

##### 允许Bean覆盖

常规场景下，即便在Bean上加了@Primary也并不能阻止Bean覆盖，因为@Primary用于同类型的Bean选择优先注入谁，属于注入阶段。而此时是Bean的名称相同，输入Bean注册阶段的问题。而SpringBoot默认是关闭了Bean覆盖的，如果出现了重复Bean，在创建Bean的时候就会报错。

常规配置就是在配置文件中配置spring:main:allow-bean-definition-overriding: true即可

但这种配置有点问题的是，如果配置交给了nacos等配置中心进行配置的话那就不会生效。因为这个配置通过nacos下发才加载的话就太慢了，Spring初始化好容器后才会拉Nacos配置，但这时候Bean已经注册好了。必须要在Spring容器创建之前，用EnvironmentPostProcessor在代码里强行开启

##### JSON序列化

###### long类型整数精度丢失

java中long类型是一个64位的整数，当这个整数传递到前端的时候通常是通过JSON格式传输的，而前端通常情况下是使用JavaScript来处理的，在JavaScript中所有的数字(包括整数和浮点数)都是以64位浮点数格式存储的。这就导致根据IEEE 754的标准，整数范围最大只能保存到2^53，所以就会导致当一个超出JavaScript精确表示范围的`long`整数从Java后端传输到前端时，如果直接作为数字类型传输，那么在JavaScript中解析这个数字时就会丢失精度。这是因为在转换过程中，超出JavaScript能精确表示的范围的部分将无法准确表示，从而导致精度丢失。

解决的办法就是使用@JsonFormat(shape = JsonFormat.Shape.STRING)注解来对实体类中的属性进行序列化和反序列化格式化的时候，将格式转为String类型，或者是在配置文件中配置

```yaml
spring:
	jackson:
		serialization:
			WRITE_NUMBERS_AS_STRINGS: true
```

###### 日期类型转换

当后端将Date类型的字段传给前端时，前端接收到的数据格式并不如我们想的那样，而是这个样子

```json
{
    "showTime": "2024-03-21T12:00:00.000+00:00",
}
```

这种格式其实是**ISO 8601字符串**，这是一种广泛使用的日期和时间表示方法，格式如`2020-01-01T12:00:00Z`。很多现代的Web框架和API在传递日期时默认使用ISO 8601格式的字符串，因为它易于解析且避免了时区的混淆。例如，Java中的`java.time`包中的`Instant.toString()`会产生这种格式。

解决方案同样可以使用@JsonFormat(pattern="yyyy-MM-dd",timezone = "GMT+8")或者是通过配置文件：

```yaml
spring：
	jackson:
		date-format: yyyy-MM-dd HH:mm:ss
		time-zone: GMT+8
```

###### 多字段报错

默认情况下@RequestBody标注的对象必须包含前端传来的所有字段。

如果没有包含前端传来的字段，就会报错：`Unrecognized field xxx , not marked as ignorable`，这是因为**MappingJacksonHttpMessageConverter**默认要求必须存在相应的字段。如果没有前台传来的某个字段，就会报错

当然也有两种解决方案

* ```java
  @JsonIgnoreProperties(ignoreUnknown = true)
  public class Test {
      
  }
  ```

* ```java
  @Configuration
  public class WebMvcConfig implements WebMvcConfigurer {
    @Bean
    public MappingJackson2HttpMessageConverter mappingJackson2HttpMessageConverter(){
  
        MappingJackson2HttpMessageConverter converter = new MappingJackson2HttpMessageConverter();
        ObjectMapper objectMapper = new ObjectMapper();
        //添加此配置
        objectMapper.configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
        converter.setObjectMapper(objectMapper);
        return converter;
    }
  }
  ```

public class JsonCustomSerializer extends BeanSerializerModifier {

	@Override
	public List<BeanPropertyWriter> changeProperties(SerializationConfig config, BeanDescription beanDesc,
			List<BeanPropertyWriter> beanProperties) {
		for (BeanPropertyWriter writer : beanProperties) {
			com.fasterxml.jackson.databind.JsonSerializer<Object> js = judgeType(writer);
			if (js != null) {
				writer.assignNullSerializer(js);
			}
		}
		return beanProperties;
	}

	public com.fasterxml.jackson.databind.JsonSerializer<Object> judgeType(BeanPropertyWriter writer) {
		JavaType javaType = writer.getType();
		Class<?> clazz = javaType.getRawClass();
		//String类型
		if (String.class.isAssignableFrom(clazz)) {
			return new com.fasterxml.jackson.databind.JsonSerializer<Object>() {
				@Override
				public void serialize(Object value, JsonGenerator gen, SerializerProvider serializers)
						throws IOException {
					gen.writeString("");
				}
			};
		}
		//数字类型
		if (Number.class.isAssignableFrom(clazz)) {
			return new com.fasterxml.jackson.databind.JsonSerializer<Object>() {
				@Override
				public void serialize(Object value, JsonGenerator gen, SerializerProvider serializers)
						throws IOException {
					gen.writeString("");
				}
			};
		}
		//Boolean类型
		if (Boolean.class.isAssignableFrom(clazz)) {
			return new com.fasterxml.jackson.databind.JsonSerializer<Object>() {
				@Override
				public void serialize(Object value, JsonGenerator gen, SerializerProvider serializers)
						throws IOException {
					gen.writeBoolean(false);
				}
			};
		}
		//Date类型
		if (java.util.Date.class.isAssignableFrom(clazz)) {
			return new com.fasterxml.jackson.databind.JsonSerializer<Object>() {
				@Override
				public void serialize(Object value, JsonGenerator gen, SerializerProvider serializers)
						throws IOException {
					gen.writeString("");
				}
			};
		}
		//DateTime类型
		if (clazz.equals(DateTime.class)) {
			return new com.fasterxml.jackson.databind.JsonSerializer<Object>() {
				@Override
				public void serialize(Object value, JsonGenerator gen, SerializerProvider serializers)
						throws IOException {
					gen.writeString("");
				}
			};
		}
		//数组、List、Set类型
		if (clazz.isArray() || clazz.equals(List.class) || clazz.equals(Set.class)) {
			return new com.fasterxml.jackson.databind.JsonSerializer<Object>() {
				@Override
				public void serialize(Object value, JsonGenerator gen, SerializerProvider serializers)
						throws IOException {
					gen.writeStartArray();
					gen.writeEndArray();
				}
			};
		}
		return null;
	}
}

##### 自动装配

我们都知道自动装配以来的是@SpringBootApplication注解里的@EnableAutoConfiguration注解，而在这个注解里最关键的则是@Import(AutoConfigurationImportSelector.class)。

> 首先需要简要介绍一下@Import注解。
>
> 这个注解的作用就是导入其他配置类或组件，他可以导入这几种类型的类
>
> | 类型                                 | 功能说明                                               |
> | ------------------------------------ | ------------------------------------------------------ |
> | 普通类（POJO）                       | Spring 会自动注册成 Bean，默认 Bean 名是类名首字母小写 |
> | @Configuration 类                    | 会像普通配置类一样被解析，里面的 @Bean 方法会生效      |
> | ImportSelector 实现类                | 可以动态选择要导入哪些 Bean 或配置类                   |
> | ImportBeanDefinitionRegistrar 实现类 | 可以手动注册任意 Bean 定义，最灵活、最底层             |
>
> 对于普通类，和@Configuration类而言，更多是比如你想注册为Bean的类不在ComponentScan的范围内，就可以使用@Import导入
>
> 而对于ImportSelector的实现类，则是可以根据实现类的逻辑，动态选择要导入那个Bean或配置而不是直接在Import中写死类名，这也是自动配置的核心。

大致流程是@SpringBootApplication注解里的@EnableAutoConfiguration里的@Import(AutoConfigurationImportSelector.class)导入了一个实现了DeferredImportSelector的ImportSelector，这个DeferredImportSelector是ImportSelector接口的子类，支持**延迟导入**。

- 延迟的好处是：**所有普通配置类先解析完，再处理自动配置**，保证自动配置能够依赖用户配置进行条件判断。

然后AutoConfigurationImportSelector里的selectImports方法就会返回所有符合条件的类的全限定名的一个String[]。

具体就是扫描“META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports”文件里列出的所有类的全限定名，然后根据**AutoConfigurationImportFilter + 条件注解** 做筛选，同时会排除用户在配置里指定的类，如：spring.autoconfigure.exclude=org.springframework.boot.autoconfigure.orm.jpa.HibernateJpaAutoConfiguration

然后调用AutoConfigurationGroup对自动配置类排序，最后吧筛选后的配置类交给Spring容器

## damai-captcha-manage-framework

该模块是对验证码的封装。分为damai-base-captcha和Damai-captcha-framework

**对于damai-base-captcha**核心的技术点就是：

1. 利用工厂模式+策略模式+模版方法模式：
   * 利用策略模式，定义验证码的一系列通用行为，以及验证码缓存的一系列通用行为
   * 利用模范方法模式，创建一个抽象类实现该接口，并将一些所有验证码通用的内容给实现
   * 最后分别继承该模版类，根据各自的特点完善内容。
   * 利用SPI服务发现机制，将接口的实现类写在META-INF/service/接口全限定名下。
   * 最后利用工厂模式，使用ServiceLoad分别获取验证码实现类和验证码缓存实现类，并分别将他们缓存进一个map中。对外提供两个不同的方法，让用户通过输入类型，获取对应的实例。
2. 然后有一个FrequencyLimitHandler以及他的实现类DefaultLimitHandler，在get，check，vertify三个方法上做限流，通过业务名+操作标识+客户端uid，使用redis来存储信息。主要是判断每个key下的value值是否超过许可，如果超过则会添加一个lock标识的key，在一定时间内拒绝操作。

关于限流的具体流程如下：

1. 首先检查客户端标识是否存在，不存在则直接放行，不做限制
2. 生成特定操作的计数键（如，“GET”，”CHECK“，“VERIFY”）
3. 生成锁定key，"LOCK"
4. 验证锁定key是否存在，若存在则代表失败次数过多，拒绝获取验证码
5. 获取当前计数，如果不存在则初始化
6. 将计数加1
7. 检查是否超过限制，超过则返回错误
8. 生成失败次数Key
9. 验证key是否存在，若不存在则通过校验，存在则判断失败次数是否大于一分钟五次，若大于则设置五分钟的Lock key

## damai-id-generator-framework

### 雪花算法

雪花算法其实就是将一个64位的long型变量分成了几个部分：符号位，时间戳，机器ID，序列号

- 第一个部分(符号位)：`1个bit`，无意义，固定为0。二进制中最高位是符号位，1表示负数，0表示正数。ID都是正整数，所以固定为0。
- 第二个部分(时间戳)：表示时间戳，精确到毫秒，2^41/(1000_60_60_24_365)=69，大概可以使用 69 年。时间戳带有自增属性。
- 第三个部分(机器ID)：表示10位的机器标识，最多支持2^10=1024个节点。此部分也可拆分成5位datacenterId和5位workerId，datacenterId表示机房ID，workerId表示机器ID。
- 第四部分(序列号)：表示序列号，同一毫秒时间戳时，通过这个递增的序列号来区分。即对于同一台机器而言，同一毫秒时间戳下，可以生成 2^12=4096 个不重复 id。

除了符号为固定占1bit，其余的时间戳，机器ID，序列号用户可以自行搭配。

* 时间戳影响的是这个分布式id能用多久，能保证多久不重复
* 机器id决定支持多少台机器部署服务
* 序列号决定在同一毫秒下的最大并发度

雪花算法的优点就是：

* 分布式环境下id不重复
* 基于时间戳，所以id是递增的
* 不依赖第三方中间件
* 算法简单，效率高

缺点则是：

* 依赖服务器时间，服务器时钟回拨时可能会生成重复 id。算法中可通过记录最后一个生成 id 时的时间戳来解决，每次生成 id 之前比较当前服务器时钟是否被回拨，避免生成重复 id。

### UidGenerator

项目里使用的雪花算法的实现来自百度开源的UidGenerator

#### BitsAllocator

首先就是这个位分配器，通过传入各部分的位偏移量来创建位分配器，用于获取相关信息，如最大workId，最大序列号等。并提供allocate方法，通过传入各部分的值，来组装成64位long类型的id。

#### WorkIdAssigner

百度的这个雪花算法的workId，是通过定义一个WorkIdAssigner接口，并抽象assignWorkerId方法。通过子类RedisDisposableWorkerIdAssigner实现对应方法来为每一台机器获取对应的workId，靠redis的increment自增对应key的value来获取。这种方式能够动态支持机器扩容，机器只要启动项目就会通过redis incr命令原子性自增，并分配其workId。

#### UidGenerator

该类是一个接口类，定义了雪花算法id的一系列相关行为，比如getUid，parseUid等

他有两个实现类，分别是DefaultUidGenerator和CachedUidGenerator。

其中CachedUidGenerator继承自DefaultUidGenerator类，拥有更完善的方法实现。这里更像是使用了模版方法的设计模式。

##### DefaultUidGenerator

首先这个类是一个模版类，实现了UidGenerator大部分方法的完整实现。重点关注一下部分：

1. 首先定义了各属性值，比如id各组成部分的位偏移量，起始时间戳等。
2. 实现了InitializingBean接口的afterPropertiesSet，当创建好Bean后为当前服务分配WorkId，WorkId就是➡️刚才提到的Redis DisposableWorkIdAssigner的assignWorkerId方法提供的。如果超过最大的WorkId会抛异常
3. nextId方法就是获取雪花ID的方法，由于是默认的雪花算法生成器，所以这里生成的雪花算法也是最基础的雪花算法。就是获取当前时间，然后和lastTimestamp比较，如果发生时钟回拨就抛异常。如果是同一秒sequence就自增并与其最大值取模运算。如果sequence等于0，说明这一秒的序列号用完了。那就会获取下一秒的序列号。如果不在同一秒序列号默认为0，然后就是记录上一秒的时间戳。把生成的时间戳和序列号交给BitsAllocator返回id

可以看到DefaultUidGenerator就是利用模版方法，提供一个最低可用版本的雪花算法。而CachedUidGenerator才是核心可用的版本

##### CachedUidGenerator

首先，他也实现了afterPropertiesSet方法，并且在原有super.afterPropertiesSet方法之后还调用了自身的initRingBuffer方法

initRingBuffer的流程主要是：

1. 通过bitsAllocator里配置的sequence的最大值即MaxSequence来决定buffer的size(因为在设计里，唤醒缓冲区就是存放当前1s内的id，那一秒里做多就能生成MaxSequence这么多的id)
2. 然后凭借这个buffersize和填充阈值，来new RingBuffer对象(这个填充阈值就是paddingFactor，他的作用就是：比如说设置为50，那么当RingBuffer里的id少于 bufferSize * paddingFactor / 100时，就会补充id
3. 创建BufferPaddingExecutor，这个类主要是负责填充的方法，以及一些定时填充任务等。如果usingSchedule为true，表示开启定时填充，可自定义定时的间隔时间，传入他的构造函数中。
   除此之外，BufferPaddingExecutor一共可以传入三个参数
   * ringbuffer，属于注意依赖
   * BufferedUidProvider，是一个函数式接口，是实际生成id的方法
   * usingSchedule，是一个boolean类型的变量，用于表示是否开启定时任务(如果为true，需要setScheduleInterval的值，该值表示，定时任务多久执行一次)
4. 然后就是为RingBuffer注入这个BufferPaddingExecutor，并且为其配置自定义的拒绝put和拒绝take的策略
5. 左后调用BufferPaddingExecutor里的paddingBuffer和start方法(这两个方法后面介绍)

除此之外，getUid方法就不再是DefaultUidGenerator那样，而是调用ringBuffer里的take方法获取。

还有就是nextIdsForOneSecond方法，这个方法是在initRingBuffer里，创建BufferPaddingExecutor时，传入的BufferedUidProvider这个函数式接口的方法实现。本质就是根据当前时间生成id。方法很简单，就是获取当前时间戳生成一个起始id。然后一个for循环加上序列号，循环长度就是最大序列号

```java
protected List<Long> nextIdsForOneSecond(long currentSecond) {
        // Initialize result list size of (max sequence + 1)
        int listSize = (int) bitsAllocator.getMaxSequence() + 1;
        List<Long> uidList = new ArrayList<>(listSize);

        // Allocate the first sequence of the second, the others can be calculated with the offset
        long firstSeqUid = bitsAllocator.allocate(currentSecond - epochSeconds, workerId, 0L);
        for (int offset = 0; offset < listSize; offset++) {
            uidList.add(firstSeqUid + offset);
        }

        return uidList;
    }
```

##### BufferPaddingExecutor

这个类就是缓冲区填充执行器。

首先，构造方法中，会获取cpu核心线程数，然后创建一个填充线程池，核心线程数和最大线程数就是2*cpu核心数。如果开启定时任务，那么会创建一个ScheduledThreadPoolExecutor，只有一个线程

现在来讲一下几个核心方法

**paddingBuffer**

这个方法就是填充缓冲区的核心方法，首先BufferPaddingExecutor类中会维护一个running属性来表示当前是否正在进行缓冲区的填充。

所以方法paddingBuffer方法中首先会调用compareAndSet通过CAS的方式尝试修改running变量，如果正在进行填充直接返回即可

正式的填充方法里，会有一个isFullRingBuffer的boolean类型变量来表示是否填充完缓冲区。

在while循环内部就是调用了BufferedUidProvider这个函数接口的provide方法获取idList，然后在for循环里一个个调用RingBuffer.put方法放入缓冲区。如果put方法返回为false，那就标记缓冲区填充完成并退出循环

然后CAS修改running变量的状态

**start**

这个方法是开启定时任务的，定时任务其实就是定时执行paddingBuffer方法填充缓冲区。该方法在CachedUidGenerator里的initRingBuffer中会最后调用。该定时任务会延迟一次执行间隔时间后才执行

**asyncPadding**

该方法是缓冲区不足时，调用的方法，会向缓冲区填充线程池提交一个线程任务，也就是paddingBuffer方法。

> 需要区分一下：
>
> 1. BufferedUidProvider是提供id的具体方法，也就是获取时间戳，然后循环生成MaxSequence那么多的idList返回
> 2. paddingBuffer是填充的具体逻辑，依赖BufferedUidProvider提供的IdList，将他们一个个调用RingBuffer.put方法塞入缓冲区
> 3. start方法是开启定时任务线程池(如果有的话)，他是定时调用paddingBuffer方法填充缓冲区，不管你容量是多少
> 4. asyncPadding方法是，在调用RingBuffer.take的时候，如果发现低于设定的阈值时，就会开启另一个线程异步调用paddingBuffer方法填充缓冲区

##### RingBuffer

实际的放入缓冲区和从缓冲区区的逻辑

首先就是缓冲区其实就是一个long[]数组slots，每个slots有两种状态CAN_PUT_FLAG和CAN_TAKE_FLAG，默认是CAN_PUT_FLAG，这两种状态代表当前slot可执行的操作。然后还有一个PaddedAtomicLong[]的数组这个是标识位flags，用来标记slots里每一个slot对应的flag状态。PaddedAtomicLong是自定义的AtomicLong的子类，为的是避免高并发时大量的cas修改导致抖动

还有两个AtomicLong的指针

| 变量     | 含义                                |
| -------- | ----------------------------------- |
| `tail`   | 已发布的最后一个 UID 序号（生产者） |
| `cursor` | 已消费的最后一个 UID 序号（消费者） |

**put**

该方法就是向缓冲区放id，被synchronised修饰。具体逻辑如下：

1. 计算首尾指针的偏移量，如果等于bufferSize - 1，说明当前是满的，调用RejectPutBuffer的拒绝策略并返回
2. 获取currentTail + 1的索引值（要对bufferSize - 1取余，因为是环形的），然后判断对应的flags数组上的状态是否是CAN_PUT_FLAG。如果不是，则调用RejectPutBuffer的拒绝策略并返回
3. 否则则将id放入对应slots位置上，并将状态设置为CAN_TAKE_FLAG。并让tail自增

**take**

该方法就是在缓冲区中取id，具体逻辑如下：

1. 获取currentCursor也就是当前已消费的最后一个UID序号，并且通过CAS的方式试图获取nextCursor，获取方式就是判断currentCursor是否等于tail，如果不想等则+1
2. 获取currentTail也就是当前发布的最后一个UID序号
3. 如果currentTail - nextCursor < 阈值，那么就调用bufferPaddingExecutor.asyncPadding();异步调用paddingBuffer方法填充缓冲区
4. 如果currentCursor = nextCursor，则表明cursor追上了tail，说明当前已经没有uid可以用了，需要调用拒绝take的策略
5. 然后再是根据nextCursor计算出索引，然后判断flag是否是CAN_TAKE_FLAG，是的话才取，然后切换flag

### SnowflakeIdGenerator

这是我们项目自己实现的雪花算法。

1. 配置workId和dataCenterId

   1. 首先workId和dataCenterId的生成使用redis lua脚本，脚本的大致逻辑就是先看是否存在snowflake_work_id_key这个键，如果不存在则初始化为0，并将snowflake_work_id_flag标记为true。然后再判断snowflake_data_center_id_key，不存在就初始化为0，然后将snowflake_data_center_id_flag标记为true。

   2. snowflake_work_id_flag和snowflake_data_center_id_flag在每次进入lua脚本时会初始化为false，两者都为true说明刚初始化好这两个key，之前并没有使用过，所以可以直接初始化好后就返回

   3. 后续的操作就是，如果workId没到最大值就+1，然后返回。workId到最大值后，就自增dataCenterId。dataCenterId也达道最大值后则将workId和dataCenterId都重置为0返回。

​	经过上述操作就初始化好了workId和dataCenterId。workId可以看成机器Id，dataCenterId可以看成机房Id

> 需要注意的是这里我们为什么要自己实现workId和dataCenterId的分配？为什么不直接使用Mybatis Plus提供的雪花算法？因为mybatis plus的实现逻辑里dataCenterId用机器的mac地址定义的，然后workId用dataCenterId定义的。而在诸如docker，k8s环境下，mac地址是可能重复的。一旦mac地址重复dataCenterId就可能重复，然后影响workId。最终影响雪花算法
>
> 其次就是，使用redis分配workId和dataCenterId能够做到机器/机房的动态扩容，机器只需要启动项目，即可由redis自动分配workId和dataCenterId

2. 然后就需要自定义一个Basis time时间戳，表示时间戳从何时开始。以及各部分的偏移量。因为之前说到：雪花算法本质就是对一个64位的long型变量做自定义bit的操作。在我的实现里：我的workId和datacenterId分别是5bit，序列号是12bit出去符号位，我的时间戳就是63-12-5-5=41bit。

   而我的整体结构为：| 符号位 1bit | 时间戳41 bit | 机房id(datacenterId) 5bit | 机器id(workId) 5bit｜序列号 12bit |

   所以各部分的偏移量：

   ```java
   		private final long sequenceBits = 12L;
       private final long workerIdShift = sequenceBits;
       private final long datacenterIdShift = sequenceBits + workerIdBits;
       
       private final long timestampLeftShift = sequenceBits + workerIdBits + datacenterIdBits;
   ```

   最后还需要准备一个变量lastTimestamp来记录上一个生成id的时间戳，这主要是用来判断服务器是否发生时钟回拨，如果	curTimestamp < lastTimestamp就是发生了时钟回拨

3. 然后就可以实现方法来生成雪花算法Id了。这里主要的变量是时间戳timestamp和序列号sequence。而这里会有三种情况讨论

   * curTimestamp < lastTimestamp，当前时间戳小于上一次生成id的时间戳时，说明发生了时钟回拨。在我们的逻辑里，首先计算出两者的差值offset，如果小于5毫秒，那就等调用wait方法等五毫秒，然后再调用SystemClock.now()方法获取一次curTimestamp，如果还是小于，那直接抛异常

   * 如果curTimestamp = lastTimestamp，说明是同一毫秒内的请求，那就必须自增sequence，自增的逻辑就是(sequence + 1) & maxSequence，如果这个值等于0，说明已经达到同一毫秒的最大值了，那就需要等到下一毫秒(这里的做法是获取当前timestamp再判断一下，如果还是等于lastTimestamp，则循环获取curTimestamp，直到大于lastTimestamp)。然后更新lastTimestamp为curTimestamp

   * 如果curTimestamp > lastTimestamp，说明不在同一毫秒或者是时钟回拨等了五毫秒后解决了。那么这时候就很简单，直接将序列号更新为随机值即可

     > 最后除了时钟回拨抛异常后，不管是什么操作都会将lastTimestamp更新为curTimestamp，如果是相同毫秒的操作则lastTimestamp不变

4. 最后，就可以调用第三步实现的方法，然后将时间戳，机器id，机房id，序列号等通过移位的方式填写

   ```java
   return ((timestamp - BASIS_TIME) << timestampLeftShift)
               | (datacenterId << datacenterIdShift)
               | (workerId << workerIdShift)
               | sequence;
   ```

#### 分库分表

我们自己设计的SnowflakeIdGenerator还有个重要的方法就是自定义orderNumber。

在我们项目中是有分库分表的逻辑的，而我们采用的是水平分库分表，也就是将部分的数据转移到其他库的其他表里。

那么水平分库分表的时候就需要决定分片键，当使用分片键来查询时，才能根据你的分片算法来定位到具体的库和表。所以也引出了一个问题：但你没用分片键来查询时，就会把你这个服务分的所有库和表拉出来一个个找，这个就叫全路由。

而在订单服务的场景下，毋庸置疑应该选择orderNumber来作为分片键，因为订单服务里最频繁的就是使用订单号来查询具体的订单。但是还有一个查询方式和他的使用频率不相上下。那就是使用userId来查询指定用户的订单。

那这时候就会有个疑问：那为什么不可以让orderNumber和userId都当作分片键？当然可以，分片键不是指某一个键，而是参与分片的键的总称。这叫联合分片，但是联合分片如果使用如ShardingSphere内置的分片算法，就得联合参加运算，少一个都得全路由。但很少情况是又用orderNumber又用userId查，大部分情况都是分开查。所以反而会适得其反

所以我们这里的做法就是：使用联合分片，但自定义分片算法。

#### 基因法

那么我们怎么才能做到用orderNumber和用userId都能查到同一个库和表呢？

首先我们需要有一个前置概念：如果我们把orderNumber和userId都转换成二进制。然后假如现在是2库4表。那么库就是0和1，表就是0，1，2，3我们需要做的就是让orderNumber和userId算出来的库和表所代表的数一致。

也就是说比如orderNumber算出来是库0表3那么userId也应该是库0表3，反之亦然。

但在解决如何算相同前，需要先解决如何算出库0表3，很简单在二进制里可以直接进行取余操作，比如库只有两个字需要一位就可以表示，表有四个，需要两位表示。那就通过 orderNumber & (tableCount - 1)，然后orderNumber >> tableGene，再计算orderNumber & (databaseCount - 1)。这样就得到了表的数和库的数，只要保证orderNumber和userId这样计算出来的值一样即可。

> 详细解析查看https://articles.zsxq.com/id_a9wxohvhts18.html

那如何保证呢？把userId后半部分植入orderNumber里不就可以了。什么意思？比如有四个库八个表，总共要表达这些库和表就需要2+3=5个bit位来表示， 那直接把userId & (5) 的结果拿到，比如是userGene，然后直接在获取哦rderNumber的时候插入

```java
return ((timestamp - BASIS_TIME) << timestampLeftShift)
            | (datacenterId << datacenterIdShift)
            | (workerId << workerIdShift)
            | (sequence << totalGeneLength)
            | userGene;
```

需要注意的是，这里userGene并不是直接覆盖原来sequence，而是将sequence左移动了userGene.length()这么长，所以sequence只是表达的数值变小了，并不会重复，而且损失的是高位。缺点就是并发减少了。

这个就是获取订单号，并植入userId基因的逻辑。很简单其实就是正常雪花算法获取timestamp以及sequence之后。根据配置的表数和库数得到对应的基因位，然后取出userId的基因，植入到orderNumber的末尾

```java
public synchronized long getOrderNumber(long userId, long tableCount, long databaseCount) {
        long timestamp = getBase();
        
        // 步骤1：计算基因位数 = 表基因位数 + 库基因位数
        // 表基因位数
        long tableGeneLength = log2N(tableCount);
        // 库基因位数
        long databaseGeneLength = log2N(databaseCount);  
        long totalGeneLength = tableGeneLength + databaseGeneLength;
        
        // 步骤2：创建基因掩码，用于提取用户ID的后N位
        long geneMask = (1L << totalGeneLength) - 1;
        
        // 步骤3：从用户ID中提取后N位作为完整基因（包含表基因+库基因）
        long userGene = userId & geneMask;
        
        // 步骤4：生成订单编号
        // 结构：[时间戳][数据中心ID][机器ID][序列号][基因信息]
        return ((timestamp - BASIS_TIME) << timestampLeftShift)
                | (datacenterId << datacenterIdShift)
                | (workerId << workerIdShift)
                | (sequence << totalGeneLength)
                | userGene;
    }
```

public class DatabaseOrderComplexGeneArithmetic implements ComplexKeysShardingAlgorithm<Long> {
    /**
     * 属性分库名
     * */
    private static final String SHARDING_COUNT_KEY_NAME = "sharding-count";
    
    /**
     * 属性分表名
     * */
    private static final String TABLE_SHARDING_COUNT_KEY_NAME = "table-sharding-count";
    
    /**
     * 分库数量
     * */
    private int shardingCount;
    
    /**
     * 分表数量
     * */
    private int tableShardingCount;
    
    @Override
    public void init(Properties props) {
        this.shardingCount = Integer.parseInt(props.getProperty(SHARDING_COUNT_KEY_NAME));
        this.tableShardingCount = Integer.parseInt(props.getProperty(TABLE_SHARDING_COUNT_KEY_NAME));
    }

    @Override
    public Collection<String> doSharding(Collection<String> allActualSplitDatabaseNames, 
                                         ComplexKeysShardingValue<Long> complexKeysShardingValue) {
        //返回的真实库名集合
        List<String> actualDatabaseNames = new ArrayList<>(allActualSplitDatabaseNames.size());
        //查询中的列名和值
        Map<String, Collection<Long>> columnNameAndShardingValuesMap = 
                complexKeysShardingValue.getColumnNameAndShardingValuesMap();
        //如果没有条件查询，那么就查所有的分表
        if (CollectionUtil.isEmpty(columnNameAndShardingValuesMap)) {
            return allActualSplitDatabaseNames;
        }
        //order_number条件的值
        Collection<Long> orderNumberValues = columnNameAndShardingValuesMap.get("order_number");
        //user_id条件的值
        Collection<Long> userIdValues = columnNameAndShardingValuesMap.get("user_id");
        
        Long value = null;
        //如果是order_number查询
        if (CollectionUtil.isNotEmpty(orderNumberValues)) {
            value = orderNumberValues.stream().findFirst()
                    .orElseThrow(() -> new DaMaiFrameException(BaseCode.ORDER_NUMBER_NOT_EXIST));
            //如果是user_id查询
        } else if (CollectionUtil.isNotEmpty(userIdValues)) {
            value = userIdValues.stream().findFirst()
                    .orElseThrow(() -> new DaMaiFrameException(BaseCode.USER_ID_NOT_EXIST));
        }
        //如果order_number或者user_id的值存在
        if (Objects.nonNull(value)) {
            //获得值后再获得实际的分库的索引
            long databaseIndex = calculateDatabaseIndex(shardingCount,value,tableShardingCount);
            String databaseIndexStr = String.valueOf(databaseIndex);
            for (String actualSplitDatabaseName : allActualSplitDatabaseNames) {
                //将所有的分库名和得到的分库索引进行匹配
                if (actualSplitDatabaseName.contains(databaseIndexStr)) {
                    actualDatabaseNames.add(actualSplitDatabaseName);
                    break;
                }
            }
            return actualDatabaseNames;
        }else {
            //如果没有分片键查询，则把所有真实库返回
            return allActualSplitDatabaseNames;
        }
    }
    
    /**
     * 计算给定表索引应分配到的数据库编号。
     * 核心思路：
     * - 分表使用ID的低位bit（最后 log2(tableCount) 位）
     * - 分库使用ID的中高位bit（跳过表基因位后的 log2(databaseCount) 位）
     * - 直接使用位运算，避免hashCode导致的分布不均
     *
     * @param databaseCount 数据库总数
     * @param splicingKey    分片键
     * @param tableCount    表总数
     * @return 分配到的数据库编号
     */
    public long calculateDatabaseIndex(Integer databaseCount, Long splicingKey, Integer tableCount) {
        // 计算表分片占用的bit位数
        long tableGeneLength = log2N(tableCount);
        
        // 将分片键右移tableGeneLength位，跳过表基因位
        // 然后与(databaseCount-1)进行按位与运算，得到库索引
        // 例如：ID=1101 (13), tableCount=4, databaseCount=4
        //   tableGeneLength=2, ID>>2=11 (3), (4-1)&3=3
        return (databaseCount - 1) & (splicingKey >> tableGeneLength);
    }
    
    public long log2N(long count) {
        return (long)(Math.log(count)/ Math.log(2));
    }
}

1. 获取在shardingsphere.yaml里配置的分库数/分表数 shardingCount
2. 因为实现了ComplexKeysShardingAlgorithm接口，需要实现doSharding方法
3. 在dosharding方法里，首先定义返回的真实库名/表名的集合 List<String> actualXXXNames
4. 通过complexKeysShardingValue.getColumnNameAndShardingValuesMap();拿到查询中的列名和值，这是一个Map，比如查询中有where id = 1 and orderNumber = 12345 and userId = 123455。那么这个map里就会有这三个键值对
5. 拿到这个map后通过get方法拿到分片键，比如这里就是orderNumber和userId
6. 如果两者都为null，直接就是返回所有库/表的名字，全路由
7. 只要有一个不为空，那就拿到value，然后算出库基因和表基因。怎么算呢？比如表基因就是(tableCount - 1) & splicingKey，库基因因为在表基因前面，所以需要分片键先向右移动tableGeneLength，所以就是(databaseCount - 1) & (splicingKey >> tableGeneLength)
8. 然后分别拿到库基因/表基因后，从所有的库名和表名里遍历找到携带包含这个基因的库/表名，然后返回，比如库基因算出来是1，那么就把1转成字符串后，匹配库名里带1的比如order_1

#### 虚拟分片

基因法虽然能很好的处理联合分片的场景，并且实现简单。就能轻松使用orderNumber或userId查询定位到同一个库表。

但有一个问题是：库表是在代码里硬编译的，即便能使用热部署能解决硬编译问题，但是如果我想进行扩容，势必就得重新计算分片位置。并且还得迁移数据库。

所以为了更好的应对这个新场景，需要有一个新的技术就是：虚拟分片

> 核心思想是在物理分片和路由规则之间，增加一层虚拟分片映射表。将原本硬编码在订单号中的物理分片信息，转换为虚拟分片ID。
>
> 即：数据 → 虚拟分片ID → 路由表 → 物理库表

简单的来讲就是，比如我把1024个虚拟分片分给8个物理片，那每一个物理分片的布局就是

```tcl
物理分片          虚拟分片范围          说明
─────────────────────────────────────────────────
物理分片0   →   虚拟分片0-127      (128个虚拟分片)
物理分片1   →   虚拟分片128-255    (128个虚拟分片)
物理分片2   →   虚拟分片256-383    (128个虚拟分片) ← 示例在这里
物理分片3   →   虚拟分片384-511    (128个虚拟分片)
物理分片4   →   虚拟分片512-639    (128个虚拟分片)
物理分片5   →   虚拟分片640-767    (128个虚拟分片)
物理分片6   →   虚拟分片768-895    (128个虚拟分片)
物理分片7   →   虚拟分片896-1023   (128个虚拟分片)
```

计算的大体逻辑就是：

1. 根据基因位确定 "大区域"（物理分片索引）-> 确定在哪个物理分片中
2. 根据模运算确定"小区域"（虚拟分片内偏移）-> 确定数据在虚拟分片中的哪一个
3. 最终 虚拟分片ID = 物理分片索引 × 128 + 虚拟偏移

> ```tcl
> 就像找一本书：
> 1. 先找书柜（物理分片索引：0-7，共8个书柜）
> 2. 再找书架（虚拟偏移：0-127，每个书柜有128个书架）
> 3. 最终位置 = 书柜号 × 128 + 书架号
> 
> 示例：
> - 书柜2（物理分片2），书架82（虚拟偏移82）
> - 最终位置 = 2 × 128 + 82 = 338号书架
> ```

需要注意的是：虚拟分片重点只是在数据库表中维护一个映射关系，是个路由表。用来确定某个分片键计算出来的Gene，对应的物理分片，最后对应的虚拟分片应该归哪个数据库表管。仅此而已。

计算物理分片的算法：
```java
/**
     * 计算物理分片索引（基于基因位）
     *
     * 算法：
     * 1. 提取表索引（后2位基因）
     * 2. 提取库索引（第3位基因，跳过表基因）
     * 3. 组合成物理分片索引（二维转一维）
     */
    private static int calculatePhysicalShardIndex(Long shardingKey) {
        // 提取表索引（低位基因）
        int tableIndex = (int) ((ORIGINAL_TABLE_COUNT - 1) & shardingKey);
        // 例如：(4-1) & 1234567890 = 3 & ...10 = 2 → 表2
        
        // 提取库索引（中位基因，跳过表基因位）
        long tableGeneLength = log2N(ORIGINAL_TABLE_COUNT);
        int databaseIndex = (int) ((ORIGINAL_DATABASE_COUNT - 1) & (shardingKey >> tableGeneLength));
        // 例如：(2-1) & (1234567890 >> 2) = 1 & ...00 = 0 → 库0
        
        // 组合成物理分片索引（二维坐标转一维索引）
        // 公式：行号(库) × 每行元素数(表数) + 列号(表)
        return databaseIndex * ORIGINAL_TABLE_COUNT + tableIndex;
        // 例如：0 × 4 + 2 = 2 → 物理分片索引2（对应damai_order_0.d_order_2）
    }
```

主要的逻辑就是：

1. 根据配置的表数目，计算表的基因
2. 越过表基因长度，计算库基因
3. 然后通过databaseGene * TableCount + tableGene即可算出对应的物理分片。

> 比如是2库4表，加入ordernumber是1234567890
>
> 1. 计算表基因：(4 - 1) & 1234567890 = 2
> 2. 计算库基因：(2 - 1) & (1234567890 >> 2) = 2
> 3. 计算物理分片：0 * 4 + 2 = 2。

再看看拿到物理分片后如何计算出虚拟分片

```java
/**
     * 计算虚拟分片ID
     *
     * 算法：
     * 1. 提取基因位（3位）→ 确定物理分片索引（0-7）→ 确定"大区域"
     * 2. 使用分片键模128 → 确定虚拟分片内偏移（0-127）→ 确定"小区域"
     * 3. 计算虚拟分片ID = 物理分片索引 × 128 + 虚拟偏移 → 最终位置
     *
     * @param shardingKey 分片键（orderNumber或userId）
     * @return 虚拟分片ID（0-1023）
     */
    public static int calculateLogicalShardId(Long shardingKey) {
        // 步骤1：提取基因位，计算物理分片索引（确定"大区域"）
        int physicalShardIndex = calculatePhysicalShardIndex(shardingKey);
        // 例如：shardingKey=1234567890，基因=010 → physicalShardIndex=2
        // 含义：数据属于物理分片2（damai_order_0.d_order_2）
        
        // 步骤2：计算每个物理分片的虚拟分片数
        // 8
        int totalPhysicalShards = ORIGINAL_DATABASE_COUNT * ORIGINAL_TABLE_COUNT;
        // 1024/8=128
        int virtualShardsPerPhysical = VIRTUAL_SHARD_COUNT / totalPhysicalShards;
        // 含义：每个物理分片细分为128个虚拟分片
        
        // 步骤3：使用分片键模128，确定在128个虚拟分片中的位置（确定"小区域"）
        int virtualOffset = (int)(Math.abs(shardingKey) % virtualShardsPerPhysical);
        // 例如：shardingKey=1234567890 % 128 = 82
        // 含义：在物理分片2的128个虚拟分片中，位于第82个
        
        // 步骤4：计算最终的虚拟分片ID（组合"大区域"和"小区域"）
        int logicalShardId = physicalShardIndex * virtualShardsPerPhysical + virtualOffset;
        // 例如：2 × 128 + 82 = 256 + 82 = 338
        // 含义：全局虚拟分片ID = 338
        //      属于物理分片2的虚拟分片范围（256-383）中的第338号
        
        return logicalShardId;
    }
```

算法步骤如下：

1. 拿到计算出的物理分片后，计算每个物理分片的虚拟分片数
2. 计算分片键在虚拟分片里的偏移量
3. 通过公式：虚拟分片 = 物理分片 * 每个物理分片的虚拟分片数 + 偏移量

> 接着上面的例子，拿到了物理分片2之后，假设总虚拟分片数是1024
>
> 1. 计算每个物理分片的虚拟分片数，这里的计算公式就是：先获取总的物理分片数，然后用总虚拟分片除以它即可
>    也就是1024 / (2 * 4) =128 128
> 2. 计算偏移量，1234567890 % 128 = 82
> 3. 计算最终分片键对应的虚拟分片：2 × 128 + 82 = 256 + 82 = 338

                          break;
                      }
                  }
              }

          }

SELECT 0 + a + b*10 + c*100 AS n

   ```
   │  需要将每张原表拆分为两张表（3类表都需要拆分）：                    │
   │  ┌────────────────────┬─────────────────┬───────────────────┐   │
   │  │ 原表               │ 保留在原表      │ 迁移到新表        │   │
   │  ├────────────────────┼─────────────────┼───────────────────┤   │
   │  │ d_order_0(0-127)   │ 0-63            │ 64-127→d_order_4  │   │
   │  │ d_order_1(128-255) │ 128-191         │ 192-255→d_order_5 │   │
   │  │ d_order_2(256-383) │ 256-319         │ 320-383→d_order_6 │   │
   │  │ d_order_3(384-511) │ 384-447         │ 448-511→d_order_7 │   │
   │  │                                                             │   │
   │  └────────────────────┴─────────────────┴───────────────────┘   │
   ```

   > 这里分数据的时候需要额外注意一件事情：需要确保**同一用户订单一致性**，什么意思呢？就是说要确保一个用户的所有订单都能定位到同一个库的同一个表里。所以这里需要用userId来计算虚拟分片
   >
   > 为什么不用orderNumber呢？不是也注入了userId的基因吗？
   >
   > 这是因为orderNumber注入userId的基因只能保证路由时能确保在一张表，但是计算虚拟分片时orderNumber%128 ≠ userId%128，如果按订单号判断，同一用户的订单会被分散到不同表。
   >
   > 但也不用担心如果用userId来分数据后，那用orderNumber查数据是不是就在别的表去了。这是不可能的，因为用orderNumber查的时候

   大致的代码流程(省略细节)

   ```java
   package com.damai.shardingsphere.virtual;
   
   import com.damai.exception.DaMaiFrameException;
   import lombok.extern.slf4j.Slf4j;
   import org.springframework.beans.factory.annotation.Autowired;
   import org.springframework.stereotype.Service;
   
   /**
    * 扩容服务
    * 
    * ⚠️ 核心修复说明：
    * 问题：订单号虽然融合了用户ID的基因位（后3位），但整体值不同，
    *      导致 orderNumber % 128 和 userId % 128 结果不同。
    *      如果按订单号的虚拟分片ID范围拆分，会导致同一用户的订单分散到不同表。
    * 
    * 解决：迁移策略改为基于用户ID的虚拟分片ID判断，保证同一用户的所有订单在同一张表。
    * 
    * 示例：
    * 用户ID:   logicalShardId = 258 (offset=2, 在前64个)
    * 订单号1:  logicalShardId = 266 (offset=10, 在前64个) ✅ 同表
    * 订单号2:  logicalShardId = 330 (offset=74, 在后64个) ✅ 如果按订单号判断会被拆分，但现在基于用户ID，不会拆分
    * 
    * 扩容后：
    * - 用户ID (258) → 保留在原表
    * - 订单号1 (266) → 保留在原表 ✅
    * - 订单号2 (330) → 保留在原表 ✅（路由映射已更新）
    */
   @Slf4j
   @Service
   public class ShardingExpansionService {
       
       @Autowired
       private VirtualShardMigrationTask virtualShardMigrationTask;
       
       /**
        * 执行完整的扩容流程
        * 场景：从2库×4表 扩展 到2库×8表
        * 
        * 迁移策略：基于用户ID的虚拟分片ID范围拆分
        */
       public void executeExpansion() {
           log.info("开始执行扩容流程：2库×4表 → 2库×8表 包含：订单表");
           log.info("⚠️ 迁移策略：基于用户ID维度，保证同一用户的所有订单在同一张表");
           
           // 第1步：创建新表（手动执行SQL）
           
           // 第2步：迁移damai_order_0库的所有表
           migrateDatabase("damai_order_0");
           
           // 第3步：迁移damai_order_1库的所有表
           migrateDatabase("damai_order_1");
           
           log.info("✅ 扩容流程执行完成");
       }
       
       /**
        * 迁移单个数据库的所有表
        */
       private void migrateDatabase(String database) {
           log.info("━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━");
           log.info("开始迁移 {} 库的所有表", database);
           
           // 1. 根据数据库名称确定虚拟分片的基础偏移量
           // damai_order_0 → 偏移量 0 (虚拟分片 0-511)
           // damai_order_1 → 偏移量 512 (虚拟分片 512-1023)
           int baseOffset = database.endsWith("_0") ? 0 : 512;
           log.info("数据库 {} 的虚拟分片基础偏移量：{}", database, baseOffset);
           
           // 2. 迁移订单表（d_order）的4张原表
           // d_order_0 → d_order_4 (虚拟分片 64-127)
           migrateSingleTable(database, "d_order_0", "d_order_4",
                   baseOffset + 64, baseOffset + 127);
           
           // d_order_1 → d_order_5 (虚拟分片 192-255)
           migrateSingleTable(database, "d_order_1", "d_order_5",
                   baseOffset + 192, baseOffset + 255);
           
           // d_order_2 → d_order_6 (虚拟分片 320-383)
           migrateSingleTable(database, "d_order_2", "d_order_6",
                   baseOffset + 320, baseOffset + 383);
           
           // d_order_3 → d_order_7 (虚拟分片 448-511)
           migrateSingleTable(database, "d_order_3", "d_order_7",
                   baseOffset + 448, baseOffset + 511);
           
           log.info("✅ {} 库迁移完成（4张表拆分为8张表）", database);
           log.info("━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━");
       }
       
       /**
        * 迁移单张表
        */
       private void migrateSingleTable(String database, String sourceTable,
                                       String targetTable, int startShardId, int endShardId) {
           log.info("开始迁移：{}.{} → {}.{}", database, sourceTable, database, targetTable);
           log.info("虚拟分片范围：{}-{}", startShardId, endShardId);
           
           try {
               // 步骤1：调用完整迁移流程（数据迁移 + 路由更新 + 缓存刷新）
               // @param startShardId 起始虚拟分片ID（包含）
               // @param endShardId 结束虚拟分片ID（包含）
               // @param database 数据库名称
               // @param sourceTable 源表名称
               // @param targetTable 目标表名称
               // @return 更新的虚拟分片映射数量
               int updatedCount = virtualShardMigrationTask.migrateVirtualShardRange(
                       startShardId, endShardId,
                       database, sourceTable,
                       database, targetTable
               );
               
               log.info("✅ {}.{} 迁移成功，更新了{}个虚拟分片映射", database, targetTable, updatedCount);
               
               // 步骤2：清理源表数据
               int deletedCount = virtualShardMigrationTask.cleanupSourceTableData(
                       database, sourceTable, startShardId, endShardId
               );
               
               log.info("✅ 源表数据清理完成，删除了{}条记录", deletedCount);
               
           } catch (Exception e) {
               log.error("❌ {}.{} 迁移失败：{}", database, sourceTable, e.getMessage(), e);
               
               // 步骤3：异常时回滚路由表
               log.warn("开始回滚路由表...");
               virtualShardMigrationTask.rollbackMigration(startShardId, endShardId, database, sourceTable);
               
               throw new DaMaiFrameException("迁移失败，已回滚", e);
           }
       }
   }

   可以看到大致的迁移流程就是

   1. 创建新库新表
   2. 对于单个库而言，将每张表按照扩容比例，把对应虚拟分片范围的数据，迁移到另一个新表中
   3. 迁移完数据后，将原表中被转移走的数据清除掉。一旦迁移出现异常报错，则直接回滚

##### 细节补充

1. 数据迁移其实就是把原表中处于[startShardId, endShardId]这个范围内虚拟分片的数据收集到一个集合中，然后批量插入新表中(具体分多少，分几次，要看扩容的大小。比如2库4表 -> 2库8表，就是多了一半的表，那就每张表分1/2出去。但如果是2库16表，那就多了4倍的表，那每张表就得分3/4的数据出去，然后分给三张新表，每张新表拿走1/4的数据)，（真实的迁移过程中，需要考虑到mysql深分页的问题，可以考虑使用游标，记录上一次查询的记录的id。并且可以分批插入，每一批1000条数据，避免长事务。且每插入一批数据的时候，最好休眠一段时间比如100ms，控制速度）
2. 关于数据验证，其实就是验证原表应该迁移到新表的数量CountA是否等于目标表的数据量CountB。如果CountA=CountB就说明数据量对了，然后再来随机抽一些数据出来看，是否一样。如果也对了，那才能大致认为数据验证成功了
3. 转移完数据后就应该更新虚拟分片的路由映射
4. 然后就是刷新物理片和虚拟分片映射的缓存
5. 最后就是随机抽取用户，验证他的订单一致性，看是否在同一个表中

> 这里对于，确保同一用户订单一致性有如下补充：
>
> 在扩容之前，如果是2库4表，总虚拟分片数1024，每个物理分片对应128个虚拟分片，库基因和表基因确定了物理分片，orderNumber或userId经过取模运算映射到物理分片时，同一个用户的订单会落在同一表上；但是扩容到如2库8表后，每个物理分片只覆盖64个虚拟分片，而老数据迁移仍按原来的取模128分片，为了兼容迁移阶段，orderNumber和userId可能映射到不同的新表，从而同一个用户的订单可能被拆到多张表。
>
> 工业上对是否保证同一个用户的订单落在同一张表主要取决于业务访问模式和查询需求：
>
> * 如果单用户访问频繁、需要事务或聚合操作方便，通常会保证同表，但扩容和迁移成本高；
> * 如果数据量巨大、主要做统计或日志分析，则可以允许用户订单跨表，从而简化扩容和负载均衡。

#### 为何传统的MOD，HASH_MOD算法会导致数据倾斜？（补充）

在传统的shardingshpere配置中，通常是选定一个分片键，然后制定分库分表的算法。

除了订单表需要基于**基因法**或者是**虚拟分片**来实现orderNumber以及userId两个分片键同时查询外，其余的表一般情况下都是选择的MOD和HASH_MOD

> MOD和HASH_MOD本质区别在于：MOD是直接拿分片键和库数或表数来进行取余运算。而HASH_MOD是先对分片键进行hash运算，然后再对库数或表数来进行取余运算。
>
> 因此，MOD适用于分片键本身就是数值类型的变量，而HASH_MOD适用于分片键是字符串类型的变量，需要先求得hash值在计算。

那么他们的写法通常就是：

```yaml
    shardingAlgorithms:
      databaseUserMobileHashModModel:
        type: HASH_MOD
        props:
          sharding-count: 2
      tableUserMobileHashMod:
        type: HASH_MOD
        props:
          sharding-count: 2
      databaseUserEmailHashModModel:
        type: HASH_MOD
        props:
          sharding-count: 2
      tableUserEmailHashMod:
        type: HASH_MOD
        props:
          sharding-count: 2
      databaseUserModModel:
        type: MOD
        props:
          sharding-count: 2
      tableUserModModel:
        type: MOD
        props:
          sharding-count: 2
      databaseTicketUserModModel:
        type: MOD
        props:
          sharding-count: 2
      tableTicketUserModModel:
        type: MOD
        props:
          sharding-count: 2
```

可以看到直接指定算法类型MOD或者是HASH_MOD，然后制定分片的数量即可。

这样就会导致的问题是：假如说userId_a % 2 = 1那么库和表拿到的都是1，也就是d_user_1.d_user_mobile_1，而如果userId_b % 2 = 0那么他拿到的库和表都是0，也就是d_user_0.d_user_mobile_0。

这导致的问题就是库0表1和库1表0永远没有数据，因为库和表拿到的路由永远是一样的。

针对这个问题，我们就需要自定义分片的算法，但这个问题本身比较简单，所以用不着像订单服务那样重新写一份分片路由的算法。

解决的方案就是**intdiv 基因位分离策略**

```yaml
// 正确的配置方式
分库: ds_${user_id % 2}                              // 使用 bit0
分表: d_ticket_user_${(user_id.intdiv(2)) % 2}        // 使用 bit1
```

 intdiv 是什么？ 

intdiv 是 Groovy 的整数除法，等价于 Java 的 Math.floorDiv() 或直接的整数除法

user_id.intdiv(2)  等价于  user_id / 2  (取整数部分)

| user_id | user_id.intdiv(2) | 计算过程  |
| ------- | ----------------- | --------- |
| 0       | 0                 | 0 ÷ 2 = 0 |
| 1       | 0                 | 1 ÷ 2 = 0 |
| 2       | 1                 | 2 ÷ 2 = 1 |
| 3       | 1                 | 3 ÷ 2 = 1 |
| 4       | 2                 | 4 ÷ 2 = 2 |
| 5       | 2                 | 5 ÷ 2 = 2 |

对于二进制而言，userId % 2其实就是提取最低位

而userId / 2或者是userId.intdiv(2)就相当于向右移动一个bit位，此时再 % 2就是提取原来倒数第二个bit位

 完整数据分布表 

| user_id | 二进制 | bit0 (id%2) | bit1 ((id/2)%2) | 分库 | 分表 | 最终路由             |
| ------- | ------ | ----------- | --------------- | ---- | ---- | -------------------- |
| 0       | 00     | 0           | 0               | ds_0 | _0   | ds_0.d_ticket_user_0 |
| 1       | 01     | 1           | 0               | ds_1 | _0   | ds_1.d_ticket_user_0 |
| 2       | 10     | 0           | 1               | ds_0 | _1   | ds_0.d_ticket_user_1 |
| 3       | 11     | 1           | 1               | ds_1 | _1   | ds_1.d_ticket_user_1 |
| 4       | 100    | 0           | 0               | ds_0 | _0   | ds_0.d_ticket_user_0 |
| 5       | 101    | 1           | 0               | ds_1 | _0   | ds_1.d_ticket_user_0 |
| 6       | 110    | 0           | 1               | ds_0 | _1   | ds_0.d_ticket_user_1 |
| 7       | 111    | 1           | 1               | ds_1 | _1   | ds_1.d_ticket_user_1 |

结果：所有4个分片都有数据，均匀分布！

​                   表_0    表_1

库_0 (ds_0)   ✔️       ✔️    ← 偶数库的奇数表和偶数表都有数据

库_1 (ds_1)    ✔️       ✔️    ← 奇数库的奇数表和偶数表都有数据

而对于字符串而言，这需要先求得其hash值，然后再用这个hash值来进行计算

```yaml
分库: ds_${Math.abs(mobile.hashCode()) % 2}
分表: d_user_mobile_${(Math.abs(mobile.hashCode()).intdiv(2)) % 2}
```

## damai-redis-tool-framework

这部分主要讲一下Redis Stream的相关用法

1. 首先定义一个消费的接口(这个是自定义的，属于扩展部分)，接口内定义了一个accept方法用于消费一个ObjectRecord。服务可执行实现具体的消费者行为
2. 定义一个Properties，配置redis stream相关的一些属性，如stream的名字，消费组名字，消费者名以及消费方式
3. 实现一个RedisStreamPushHandler用来封装push消息的操作，生产者直接调用对应的push方法即可往服务的properties里配置的stream发送消息
4. 然后则是消费者配置，前面的三部其实都是扩展(扩展不同消费者的accept方法，封装push方法，封装redis stream的属性配置)。这里的消费者则需要根据要求来，实现StreamListener接口，重写onMessage方法。主要是从message中获取messageId和消息体，然后将消息体传入第一步里，接口的实例所实现的accept方法里
5. 然后则是定义stream和消费者组的绑定关系，首先redis stream无法为未创建的stream配置消费者组，且没有创建stream的方法。需要先发送一条消息，然后会自动创建，再绑定group，最后删掉刚才的消息即可
6. 最关键的一步，配置StreamMessageListenerContainer。
   1. 首先先创建配置选项options，比如异常处理方法，线程池，消息序列化，消息监听的超时时间等等
   2. 用这个options和RedisConnectFactory创建ListenerContainer
   3. 验证配置文件里配置的消费方式（广播还是消费者组，广播就是所有消费者都能收到消息，消息可以被消费多次。而消费者组则是，消费者争夺消息的消费权，只能被消费一次）
   4. 根据配置文件里配置的消费方式来进一步配置
      * 如果是消费者组，则创建stream，并绑定消费者组。然后配置自动ACK，避免消息堆积在PEL中(PEL是 **消费者组内部的一个状态记录**，用于跟踪 **哪些消息已经被分配给消费者，但还没有被 ACK**。)，然后重点是StreamOffset这个消息消费的偏移量，设置为ReadOffset.lastConsumed()，也就是从上一次消费的地方开始读消息
      * 如果是广播，则配置成StreamOffset.fromStart，从头开始消费

### 布隆过滤器

布隆过滤器的使用很简单：

1. 定义一个Properties，添加name 名称，expectedInsertions 预计容量 ，falseProbabilitys 误判率，这三个属性即可
2. 随后创建一个BloomFilterHandler，定义一个RBloomFilter类型的成员变量即可，后用就是通过操作他来add，contains等。并在构造函数中注入RedissonClient以及BoolomFilterProperties。直接通过redissonClient.getBoolmFilter填入对应的参数即可得到RBloomFilter，再把它注入给成员变量即可。然后对外暴露add，contains，getSize等方法
3. 最后将这个BloomFilterHandler配置成Bean就大功告成

重点是理解布隆过滤器的原理，布隆过滤器本质上就是使用一个位数组和k个独立的hash函数，通过将元素经过k个hash函数映射到位图的k个位置上，并标志为1。后续使用contains判断元素是否存在时，就是将元素经过k个hash函数计算后判断位数组里这k个位置的值。都为1代表元素**可能存在**，只要有一个不为1就代表元素**一定不存在**

布隆过滤器只能新增不允许删除。因为位数组里的某个位可能是多个元素共同设置的。比如元素 A 和元素 B 的某个哈希值都指向位置 5，删除 A 的时候如果把位置 5 置 0，那查询 B 的时候就会误判为不存在。想支持删除要用 Counting Bloom Filter，把每个位置换成计数器，但空间会膨胀 4-8 倍。

布隆过滤器的误判率取决于三个因素：位数组大小m，哈希函数个数k，已插入的元素数量n

理论上的最优哈希函数个数是 `k = (m/n) * ln2`，大约是 0.7 倍的位数组与元素数量之比。

实际工程中，一般这样估算：

1）要求误判率 1%，每个元素大约需要 10 bit

2）要求误判率 0.1%，每个元素大约需要 15 bit

3）要求误判率 0.01%，每个元素大约需要 20 bit

比如要存 1000 万个元素，误判率控制在 1%，位数组大约需要 1 亿 bit，也就是 12MB 左右。

> 因此布隆过滤器的误判率不可能降低到0，因为hash冲突只能限制但无法避免，要想降低误判率，就得增加位数组大小以及hash函数的个数，但永远不可能为0。布隆过滤器本质就是用少量的空间代价来换取“肯定不存在”的确定性结果

### Redisson公共配置

这里其实没多少说的，主要是就是配置Redisson客户端，并且有一个本地的锁缓存。

最主要的就是锁信息工厂LockInfoHandleFactory

这个锁信息工厂就是为后续的分布式锁和幂等组件作准备，主要是根据参数拼接锁的名字。而LockInfoHandleFactory就是根据提供的LockInfoType来获取对应的LockInfoHandle。

LockInfoHandle接口提供了两个方法getLockName和simpleGetLockName。前者用于分布式锁的注解使用，后者用于方法类的分布式锁使用

他分别就有三个实现类，AbstractLockInfoHandle，ServiceLockInfoHandle和RepeatLimitLockInfoHandle。

其中AbstractLockInfoHandle利用模版方法，为ServiceLockInfoHandle和RepeatLimitLockInfoHandle提供了公共的拼接实现。使得ServiceLockInfoHandle和RepeatLimitLockInfoHandle只需要各自提供自己独立的名称即可，比如分布式锁就是SERVICE_LOCK，幂等就是REPEAT_EXECUTE_LIMIT。双方通过重写父类提供的getLockPrefixName方法，传入自己的名称就能完成拼接

具体的getLockName方法的实现其实就是：项目名+锁类型+锁名+传入的key拼接而成，其中锁名和keys由业务中实际场景提供。比如：@ServiceLock(lockType= LockType.Write,name = SEAT_LOCK,keys = {"#programId","#ticketCategoryId"})，这里name就是SEAT_LOCK，keys就是programId和ticketCategoryId的**值**。

这里的难点就在于如何把形如“#programId”这样的表达式替换成真实的值，对于使用工具类方法实现的分布式锁中，就是直接传入keys的具体值，这时直接存入即可然后String.join即可。

对于注解来说，此时就需要使用SpEL来解析表达式。

> 当然对于简单的keys，比如就是单一字段的值，那么其实可以直接通过joinPoint分别获取参数值和参数名来匹配表达式自行解析。好处就是不需要使用SpEL，学习成本低，足够简单。
>
> 但这里为了标准化，且为了日后的扩展如：三元运算符，集合，对象属性等场景，选择使用SpEL来解析

其实SpEL解析表达式很简单，主要是就是需要有ParameterNameDiscoverer参数名发现和ExpressionParser表达式解析两个即可。

总的来说就是传入method方法本身，parameterValues参数值，definitionKeys表达式。其中method和parameterValues通过joinPoint获取。然后就是创建EvaluationContext，传入ParameterNameDiscoverer，最后通过ExpressionParser来判断获取key对应的值

### ServiceLock

分布式锁组件的设计较为简单。

首先就是继承AbstractLockInfoHandle实现ServiceLockInfoHandle，重写getLockPrefixName方法，提供分布式锁的锁前缀字段

然后定义分布式锁统一接口ServiceLocker，定义分布锁的一系列能力，如tryLock，getLock，lock，unlock等。

分布式锁总共提供了四种类型的锁，分别是ReentrantLock，WriteLock，ReadLock，FairLock即可重入锁，写锁，读锁，公平锁等。（注意这里的ReentranLock不是JUC提供的可重入锁）所以会分别实现对应的功能，其实大部分内容都一致，就是从redissonClient中获取

实现好所有的锁实现后，就定义一个缓存ManageLocker并注册成bean，调用构造函数完成初始化的时候，就会提前将所有类型的分布式锁new出来，然后放到一个HashMap中。并对外提供四个接口来获取四种不同的锁

有了策略模式和统一对外提供锁对象的类之后，就需要有一个工厂类，来隐藏内部细节，对外提供对象。ServiceLockFactory内部注入了ManageLocker，只需要提供LockType就可以得到对应类型的锁对象

做完这些准备工作之后，ServiceLock提供了两种操作的方式。

1. 基于ServiceLock注解
2. 基于ServiceLockTool的方法

#### ServiceLockTool

这是利用ServiceLockFactory获取相应类型的锁（未提供的话默认就是可重入锁），然后封装了一系列方法。

而在ServiceLockTool中还提供了两种上说模式：

1. 通过getLock，获取到锁对象，然后自行控制上锁和解锁实际，灵活性更高。
2. 通过包装TaskCall和TaskRun任务调用execute/submit方法对任务上锁。这种方式更简单直接，只需要提供任务即可，无需操心上锁解锁时机。

对于execute/submit方法中：

* execute方法负责TaskRun方法，TaskRun是项目自己定义的一个FunctionInterface函数式接口，可以用lambda直接传入，也可以包装好的方法。他是处理没有返回值的任务
* submit和execute正好相反，他负责TaskCall方法，TaskCall也是一个函数式接口，但他处理的是有返回值的任务

#### ServiceLock Annotation

分布式锁注解有以下参数：

* LockType锁类型，默认可重入锁
* name业务名称，默认""，用于拼接锁名
* keys自定义业务键，用于拼接锁名
* waitTime加锁失败的等待时间，默认10
* TimeUnit时间单位，默认TimeUnit.SECONDS
* lockTimeoutStrategy加锁超时处理策略，默认打印错误，并抛出RunTimeException
* customLockTimeoutStragegy自定义加锁超时策略，默认为""（如果有，需要写上注解所在类的一个方法名作为自定义超时策略，并且该方法的参数需要和该注解方法一致，在切面里会通过joinPoint得到具体的方法并执行）

切面的具体逻辑：
首先切面定义为@Around("@annotation(servicelock)")

1. 通过LockInfoHandleFactory拿到分布式锁的LockInfoHandle，并传入joinPoint，name，keys基于SpEL拼接锁名
2. 然后再拿到lockType，waitTime，timeUnit这些参数值。通过ServiceLockFactory拿到对应的锁对象，然后待用tryLock方法尝试获取锁
3. 如果获取成功则调用joinPoint.proceed()执行业务方法，并在finally解锁
4. 如果获取锁失败，则判断用户是否提供自定义失败策略。
5. 如果未提供，则调用默认的失败策略
6. 如果提供了，则通过joinPoint获取当前方法method以及当前对象Target。通过target对象拿到对应的类，通过getDeclaredMethod传入customLockTimeoutStrategy和method.getParameterTypes。然后通过joinPoint获取到args参数值，通过invoke方法调用对应方法

### RepeatExecuteLimit

幂等只有注解的实现，有如下相关参数：

* name业务名称，用于拼接锁名，默认为""
* keys自定义业务键，用于拼接锁名
* durationTime保持幂等的时间，默认为0
* message当消息执行已经出发防重复执行的限制时，提示信息，默认为"提交频繁，请稍后重试"

具体的切面逻辑为：

1. 通过LockInfoHandleFactory获取幂等的LockInfoHandle，并提供LockInfoHandle获取对应的lockName，并在前拼接repeat_flag得到时机操作的repeatFlagName
2. 判断redis中repeatFlagName对应的value是否为SUCCESS_FLAG，如果是的话说明该业务已经执行成功了，直接抛异常，异常信息就是message
3. 如果不是SUCCESS_FLAG，则从LocalLockCache中获取本地锁，并尝试上锁。（这一步主要是利用本地锁减少部分分布式锁竞争的量）
4. 如果获取锁失败直接抛异常，如果获取锁成功，则通过ServiceLockFactory获取公平锁，并尝试tryLock上锁。如果分布式锁上锁失败也直接抛异常。
5. 如果加锁成功，则再检查一次repeatFlagName对应的value是否是SUCEESS_FLAG，如果是的话就抛异常（这里是利用了双重检验的思想）。如果不是就正常执行业务，任务执行完后，如果设置了durationTime，则将对应的repeatFlagName的value设置为SUCCESS_FLAG，过期时间就是durationTime。
6. 最后先解锁分布式锁，再解锁本地锁

> 这里需要额外补充，该幂等注解是可以不设置durationTime的，如果不设置durationTime那么就是并发幂等。在同一时刻下只有一线程能执行成功，并利用本地锁和分布式锁双层锁快速失败，提高性能。（注意这种方式只能防止同一时刻的并发幂等，如果有间隔时间那么并不能防止）
>
> 如果设置了durationTime就是时间幂等，可以防止durationTime时间内该方法的执行是幂等的

## damai-service-delay-queue

该模块主要是使用Redisson提供的延迟队列，采取分片的思想，提高并发。

其实核心无非就是生产者 -> 分片 -> 延迟队列 -> 到时间将消息加入阻塞队列 -> 生产者监听阻塞队列获取到任务并执行对应逻辑。

整个流程就是这样的。但过程中做了很多封装：

1. 首先定义了DelayQueueProperties，配置了一些延迟队列的属性，里面主要包含消费者消费任务的线程池配置，以及一个关键属性isolationRegionCount，也就是分片数。
2. 有了分片数就有决定分片的路由，对应的就是IsolationRegionSelector分片选择器，他内部有一个AtomicInteger这个就是计数器，通过CAS原子性地递增计数器，但计数器>=分片数时就会重置为0，使得任务能够均匀地分摊在每一个队列上。对外提供index方法返回当前分片数
3. DelayQueueProduceCombine是对IsolationRegionSelector的上层封装，IsolationRegionSelector只负责告诉现在轮到第几个分片了，实际执行路由的是DelayQueueProduceCombine。他内部通过一个List存储了同一个topic下的延迟队列分片，在构造函数中根据分片数初始化分片选择器，并将对应的分片延迟队列加入到List中。对外提供offer方法通过调用IsolationRegionSelector.index()方法获取当前的分片数，从List中获取对应分片索引的延迟队列生产者，并调用其内部的offer方法完成路由。(这里需要补充的是，topic是逻辑上区分业务的主题，真实的分片主题是topic + "-" + index拼接的，比如orderCancel-1代表取消订单的第二个分片)
4. DelayQueueContext则是最上层的上下文封装，对外报漏sendMessage方法发送延迟消息。其内部依靠一个ConcurrentHashMap，缓存不同topic下的DelayQueueProduceCombine，调用sendMessage方法的时候通过传入的topic获取DelayQueueProduceCombine，然后DelayQueueProduceCombine通过IsolationRegionSelector获取分片数，最后调用生产者的offer方法发送消息
5. DelayBaseQueue就是延迟基础队列，生产者和消费者都需要继承他。他主要是提供RedissonClinet客户端和RBlockingQueue的bean，并在构造函数中通过传入的分片topic初始化阻塞队列（redisson的延迟队列基于阻塞队列，构造延迟队列需要传入对应的阻塞队列）
6. DelayProduceQueue就是对应的生产者，负责利用父类构造的阻塞队列创建延迟队列，并绑定分片主题。对外提供offer方法发送消息
7. ConsumerTask是消费者接口，需要使用延迟队列的业务需要自己实现该接口提供execute消费任务的方法以及topic主题
8. DelayConsumerQueue是对应的消费者监听，构造方法中通过传入消费者实例，配置文件以及分片主题。分别创建监听线程池和消费线程池，其中监听线程池只有一个线程。提供listenStart()方法开始监听任务，该方法被synchrnized修饰，确保消费任务原子性执行。大致逻辑就是监听到任务后从阻塞队列里取出来开启消费线程，调用消费者实现的execute方法消费任务
9. DelayQueueInitHandler时延迟队列初始化的启动器，实现了ApplicationStartedEvent，通过监听应用启动事件，当应用启动时获取所有实现了ConsumerTask的实现类，遍历每一个ConsumerTask实现类，获取他们的分片数，创建对应的监听者来监听对应分片主题下的阻塞队列。

上述是整个延迟队列组件的实现，下面不会讲解每个类的具体作用，而是按照逻辑，讲解项目启动时，这些类都做了什么。

1. DelayQueueInitHandler通过监听ApplicationStartedEvent，获取所有实现ConsumerTask接口的类，并循环便利这些实现类，获取每一个实现类的分片数，并创建对应数量的监听者监听对应分片主题下的阻塞队列，并且调用listenstart方法开启监听
2. 这些监听者在创建之初会分别创建监听线程池和消费线程池，在开启监听后会为每一个分片主题开启一个监听线程池，一旦监听到阻塞队列出现任务，这会开启消费线程池调用对应ConsumerTask实现类的execute方法消费任务
3. 项目中当通过DelayQueueContext提供的sendMessage方法发送延迟消息时，会通过传入topic，判断当前的map是否存在对应topic的DelayQueueProduceCombine，如果没有的话会创建并返回。调用DelayQueueProduceCombine提供的offer方法发送消息
4. 在创建DelayQueueProduceCombine时，会根据传入的配置文件获取分片数n，并且创建n个DelayProduceQueue，通过topic + "-" + i来作为实际的主题绑定延迟队列，存放到List中
5. 调用DelayQueueProduceCombine里的offer时，会通过isolationRegionSelector拿到分片数，这在isolationRegionSelector内部主要是维护一个AtomicInteger，只要当前值小于分片数就自增返回，大于等于时就重置为0开始。拿到分片数后从List中取出对应分片索引下的DelayProduceQueue，并调用他的offer方法正式发送消息

## damai-thread-pool-framework

该模块主要是提供线程任务包装的方式，借鉴TTL的思想，使得traceId等存储在threadLocal中的数据能够在线程池中传输，确保链路不会中断。

核心部分就是：

* BaseThreadPool，这是线程池基类，提供了wrapTask方法可以包装Runnable任务和Callable任务。整个包装的思想其实就是把真实的Runnable/Callable任务再包装一层Runnable/Callable任务。即比如说现在有个任务Runnable r，线程池executor。之前执行任务都是executor.execute(r)然后就会启动一个线程去执行任务。但现在我们通过吧Runnable r任务再包装一层任务，即Runnable wrapTask(Runnable r)。第一层Runnble任务是主要的线程任务，第二层Runnable其实就是解决线程池线程无法携带traceId的问题。

  1. 在wrapTask中需要传入三个参数，Runnable/Callable任务，父线程MDC Context，父线程ThreadLocal Context
  2. 传入后通过preprocess方法的逻辑：
     1. 首先分别使用holdContext和mdcContext两个map保存线程池线程的上下文
     2. 依次判断父线程的mdc/threadLocal上下文是否为空，如果不会空则替换给线程池线程。
     3. 将线程池线程的mdc和ThreadLocal上下文以“holdContext”和“mdcContext”作为key区分存放到一个Map<String,Map<String,String>>类型的map中返回
  3. 从返回的Map<String,Map<String,String>> map中取出线程池线程的mdc和threadLocal上下文
  4. 调用task.call()方法执行第一层线程任务
  5. 任务执行用try-catch包裹，并在finally中加入postprocess逻辑
     1. 主要就是清理线程池线程的mdc和threadLocal上下文，并替换回他执行task.call之前mdc和threadLocal上下文

  上述就是整个包装任务的逻辑，可以看到其实就是在一个Runnable任务的基础上再包装了一层Runnable任务，包装的这层Runnable任务其实就是在原始任务前替换线程池线程的上下文，任务执行后再把上下文替换回去

* 而在BusinessThreadPool中则是即成了BaseThreadPool并创建了线程池。还提供了两个适配新版包装任务的Runnable和Callable任务的execute方法

## damai-spring-cloud-framework

### damai-service-component

该模块主要为各服务如order，program，user提供Filter来传递traceId。

然后针对服务间调用也就是openfeign调用无法传递traceId的情况做了优化，加入了interceptor来帮助解决

这里有个很误导的点在于为什么有了处理网关->服务的traceId，为什么服务间的这个就失效了，原因在于：

1. **网关本身就是 HTTP 服务端 + HTTP 客户端**，Filter 正好能覆盖到请求进入和请求转发。
2. 服务间的调用本质是**A 主动发起一个新的 HTTP 客户端请求**。**Filter 并不会参与你作为客户端发请求的过程**，它只管“别人调我”。所以这时 traceId 必须靠 **客户端拦截器（Interceptor）** 来传递。

总之：**Filter 作用域**：只负责处理 **进入本服务的请求**（服务端逻辑）。**服务间调用传递 traceId**：必须在 **客户端调用拦截器**里把 traceId 从 `MDC/ThreadLocal` 取出来，加到请求头，再传递给下游服务。

 所以链路追踪要“两头结合”：

- **服务端入口**：Filter 把请求头里的 traceId 放到 MDC。
- **客户端出口**：Interceptor 把 MDC 里的 traceId 塞到请求头。

用大白话讲就是：Filter只能管进入A服务的请求，但管不了A服务发出去的请求，B服务确实能处理A服务发的请求，但由于A服务发出去的请求没经过处理，发出的请求是没有traceId等信息的。所以需要interceptor来拦截发出去的请求，从当前线程的上下文中拿到之前的请求信息，然后放入当前请求中。这样B的Filter处理时就能正确使用traceId。

> 讲了interceptor和filter的差别后，服务间调用无法传入traceId的主要原因是，发送请求的代码由RestTemplate等框架实现，业务层并没有对traceId做适配，不像网关，生成了traceId就加入到http header中。服务间调用没这一步。所以需要定义一个拦截器interceptor，在请求发送前拦截住，注入traceId再发出去。

**FeignRequestInterceptor**实现服务间调用传递traceId的主要流程：

1. 从RequestContextHolder中获取到当前服务正在处理的“入站请求”
2. 从“入站请求”中取出traceId
3. 添加到RequestTemplate的header中

> RequestTemplate是Feign 用来“描述一个即将要发送的 HTTP 请求”的模型对象

**BaseParameterFilter**就是业务服务过滤器，为order，program，user等业务服务传递traceId。实现方式很简单，直接从header中获取，然后存入MDC以及ThreadLocal中

除此之外还与一个请求包装类CustomizeRequestWrapper，他继承了HttpServletRequestWrapper，内部有一个成员变量requestBody重来存放请求体。并提供RequestWrapperFilter，请求进来时替换成CustomizeRequestWrapper并保存请求的请求体，允许请求体反复使用

> CustomizeRequestWrapper的实现原因：Servlet 规范里一个非常关键但坑人的点，`HttpServletRequest.getInputStream()` 只能读一次原因是：request body 本质是一个流（Stream）一旦被读完，流就“空了”。
>
> 一个很经典的使用场景就是：请求在到达controller前，在filter中读了请求，并进行了写日志/签名/解密等业务时，如果controller再读body就会报错。
>
> 所以这里直接一次把 request body 全部读出来，存成 String，后续每次有人再读 InputStream 时，我就“重新造一个流”给他。
>
> 核心实现就是在构造函数中：requestBody = StringUtil.inputStreamConvertString(request.getInputStream());真正从网络流中把 body 读完，缓存成一个 String。然后重写getInputStream()方法，通过ByteArrayInputStream byteArrayInputStream = new ByteArrayInputStream(requestBody.getBytes(StandardCharsets.UTF_8));手动制造一个流出来。
>
> 

### damai-service-initialize

该模块主要是提供服务统一的初始化任务。将四种初始化方式进行了封装，通过模板模式和策略模式，使用时只需继承对应的类然后实现对应的初始化顺序和初始化逻辑，就能按照顺序在服务启动时自动执行初始化。优点是方便统一管理初始化的任务，且能保证同一种初始化方案之中的初始化顺序。

还提供了组合模式，用于构建和执行具有树结构的业务逻辑

#### InitializeHandler

该类是初始化执行的顶级抽象接口，主要有三个抽象方法：

* type()，初始化执行的类型
* executeOrder()，初始化执行的顺序
* executeInit()，初始化执行的具体逻辑

他也有四个实现类：

1. AbstractApplicationCommandLineRunnerHandler,用于处理 {@link CommandLineRunner} 类型 初始化执行 抽象
2. AbstractApplicationInitializingBeanHandler,用于处理 {@link InitializingBean} 类型 初始化执行 抽象
3. AbstractApplicationPostConstructHandler,用于处理 {@link PostConstruct} 类型 初始化执行 抽象
4. AbstractApplicationStartEventListenerHandler,用于处理 {@link ApplicationStartedEvent} 类型 初始化执行 抽象

这几个实现类利用模版的的方式为子类提供type属性，子类只需要专注于定义executeOrder执行顺序，以及executeInit初始化逻辑。

但现在有个问题，我理解的初始化逻辑肯定得是：比如我现在有@PostConstruct相关的三个初始化任务A，B，C我希望的肯定是有个统一的启动初始化任务的方式他就能帮我把A，B，C三个初始化任务按顺序执行。现在A，B，C有了顺序，有了初始化逻辑，但是唯独缺少了一个统一的**发现**以及**启动**的executor

#### AbstractApplicationExecute

因此在上述的需求上，产生了AbstractApplicationExecute，他从应用启动上下文中拿到了所有InitializeHandler的实现类，并缓存在map中

通过stream流，按不同初始化类型分类并按照各自的executeOrder排序，然后对于每一个type，按顺序调用他们的executeInit执行初始化逻辑
```java
public void execute(){
        Map<String, InitializeHandler> initializeHandlerMap = applicationContext.getBeansOfType(InitializeHandler.class);
        initializeHandlerMap.values()
                .stream()
                .filter(initializeHandler -> initializeHandler.type().equals(type()))
                .sorted(Comparator.comparingInt(InitializeHandler::executeOrder))
                .forEach(initializeHandler -> {
                    initializeHandler.executeInit(applicationContext);
                });
    }
```

看到这里我们实现了一个统一的管理器，他可以按照type过滤，并按照executeOrder排序执行初始化逻辑。但还有个问题就是**谁来调用这个execute方法执行逻辑？**

所以很容易想到：为每一个type设计一个启动器

* ApplicationCommandLineRunnerExecute用于处理 {@link CommandLineRunner} 应用程序启动事件。
* ApplicationInitializingBeanExecute用于处理 {@link InitializingBean} 应用程序启动事件。
* ApplicationPostConstructExecute用于处理 {@link PostConstruct} 应用程序启动事件。
* ApplicationStartEventListenerExecute用于处理 {@link ApplicationStartedEvent} 应用程序启动事件。

各自重写type方法，填写的type一定得和InitializeHandler实现类对应的type一致

这样每个启动器，在处理不用的初始化任务时，但实际一到，就会调用AbstractApplicationExecute.execute()，扫描出所有InitializeHandler实现类中与他们type相同的类，并按照executeOrder的顺序执行初始化逻辑

#### 组合模式

组合模式的实现基于AbstractApplicationStartEventListenerHandler，在ApplicationStartEvent事件发布时，就会执行executeInit方法，方法内部调用的是compositeContainer.init(context);

首先AbstractComposite是组合模式的抽象类，定义了组合模式的一系列行为：

1. 子节点列表
2. 具体的业务逻辑
3. 组合模式的类型
4. 父节点执行顺序
5. 当前组件的执行层级
6. 当前组件在同一层级中的执行顺序
7. add方法，添加子节点
8. allExecute方法，由root节点调用，按照设定的层次结构执行每个组件的业务逻辑

**CompositeContainer**就是组合模式的执行控制器，他的逻辑其实很简单

1. 在初始化的时候，通过应用上下文拿到所有AbstractComposite的实现类
2. 并让他们按照type属性分组，存放在Map<String, List<AbstractComposite>>这样的map中
3. 然后遍历每一个map，对每一个type下的实现类，按照他们的一些属性如：父类执行顺序，自身层级，自身在同层的执行顺序等调用build方法将他们拼装成树型结构。并返回根节点。
4. 如果根节点不为空，则存放在Map<String, AbstractComposite>这样的map中。后续只需要通过type就可以拿到该类型下树状任务的根节点，并执行executeAll方法。

现在讲一下是如何将这些同类型下的类，封装成树状结构的：

1. 首先是对Map<String, List<AbstractComposite>>这个map进行的forEach，并向build方法传入value。所以build方法传入的应该是List<AbstractComposite>，即同类型下所有的AbstractComposite实现类
2. 在build方法中还有一个map，叫做groupedByTier，类型为Map<Integer, Map<Integer, AbstractComposite>>，第一个Integer代表层级，第二个map里的Integer代表当前对应的AbstractComposite在第一个Integer层级里的执行顺序。
3. 然后在build方法中就开始遍历传入的List<AbstractComposite>，然后判断当前是否存在AbstracteComposite.executeTier()层级的key，如果不存在则创建，存在的话则直接将AbstractComposite.executeOrder, AbstractComposite插入到该层级对应的map中
4. 然后拿到groupedByTier里最小的层级，传入buildTree方法中正式开始构造key
5. 执行完BuildTree方法后，groupedByTier相当于已经建立好联系了，直接返回最小层级对应的AbstractComposite返回。
6. 然后对于最开始的初始化方法中，拿到build返回的根节点后，就将根节点加入到内存中缓存起来。要执行组合任务的时候，直接通过类型获取根节点执行executeAll方法即可

下面在讲一下BuildTree方法是如何构造的

1. 首先创建了两个map，一个是currentLevelComponents，另一个是nextLevelComponents，用来构造树节点的父子关系。currentLevelComponents初始化为groupedByTier里根节点，nextLevelComponents初始化为根节点层级+1的这些节点
2. 循环遍历nextLevelComponents的value，获得他们的父类执行顺序，从currentLevelComponents中取出对应执行顺序的父节点，并把这个子节点添加到父节点的子节点list里
3. 每执行完一层级，则递归调用buildTree，直到获取的currentLevelComponents为null时返回

其实executeAll方法的流程也很简单，只是简单的层级遍历顺序执行。

下面就是定制化任务最主要的实现：**消息记录对账**

  消息对账有两个表，消息发送表和消息消费表。消息对账的核心逻辑其实很简单：

1. 发送消息之前先创建消息发送记录表
2. 将发送表记录插入数据库后在正式发送消息，发送消息的代码用try-catch包装
3. 如果发送成功则将消息发送表的消息发送状态修改为发送成功
4. 如果捕获了异常，则在catch中将发送消息的状态修改为发送失败

与之对应的，消费消息的时候也是一样

1. 消费消息之前先通过消息id查询是否存在消息消费记录，如果消息消费记录存在并且消费状态为消费成功则直接返回
2. 那么此时就有两种情况，消息消费过但消息消费失败了，还有就是这是消息第一次消费
3. 如果是第一次消费，那就在消费前插入一条消息消费的记录。如果消息插入失败则直接返回
4. 无论是第一次消费还是之前消费失败了，到了这一步都会记录下此时的消费记录id和消费次数（如果之前消费过，那么消费次数会+1）
5. 此时正式消费消息，如果消费成功，修改状态为消费成功，否则为消费失败，如果是业务异常则表明消费成功，如果是其他异常则消费失败

至此我们知道了大致的工作逻辑，那下面来看看是如何实现的。

需要注意的是，消息的记录有一个字段就是MessageType消息类型，所以我们的实现里也采用平时常用的模式：

会有一个接口，定义了消息对账的完整行为，不同的消息类型只需要去实现该接口的方法即可。随后会有一个上下文Context，该Context内部会有一个Map，他的key是消息类型，value则是对应的消息对账实现类。并对外提供get方法，传入对应的消息类型即可获得对应的消息对账实现类。然后便可以使用已实现的方法来处理各自消息的对账任务。

所以在这里，ExceptionMessageHandler则是消息对账的行为接口，他定义了三个方法分别是：1. 获取没有对账的消息发送记录集合 2. 处理对账（延迟订单里的实现则是：再消费一次消息）3. 消息类型

于此同时ExceptionMessageHandlerContext则是工厂类，会将所有ExceptionMessageHandler实现类作为List注入进去。并且存放在Map中，对外提供getExceptionMessageHandler方法传入MessageType获取对应ExceptionMessageHandler

而在我们项目中ExceptionMessageHandler的实现类则是与延迟队列相关的DelayOrderCancelExceptionMessageHandler

* 他实现的第一个方法“获取没有对账的消息发送记录集合”逻辑其实很简单，其实就是查出**所有未对账成功，且发送时间小于（当前时间-延迟时间-10秒）并且消息类型为DELAY_ORDER_CANCEL的消息记录**
* 第二个方法“处理对账”，则是对每一个MessageProducerRecord，重新向延迟队列再发送一次一样的订单取消的消息，如果发送成功则发送状态为发送成功，反之为发送失败
* 第三个方法消息类型，很显然就是**DELAY_ORDER_CANCEL**

有了工厂类和实现类之后，就是具体算法的实现类，在这里其实就是利用定时任务，定时调用第一个方法查到对账失败的记录，然后对每个记录都调用handle方法重新发送一次立刻执行的延迟消息

首先ReconciliationTask则是对账任务执行的接口，定义了run方法，这是一个函数式接口。用于做任务的包装

然后就是ReconciliationTaskQueue，他是对账任务队列，内部定义了从事次数，以及一个存放任务的阻塞队列。内部定义了私有线程池，以及一个外部线程池。提供了一个业务方调用的方法putTask，将ReconciliationTask存放到阻塞队列中。

外部线程池负责监听阻塞队列，并从阻塞队列中将ReconciliationTask取出来。而内部线程池则负责执行ReconciliationTask。ReconciliationTask会被放入一个executeWithRetry方法中执行，在executeWithRetry内，ReconciliationTask只允许失败重试3次，每次重试会休眠1s，超过最大重试次数则会打断线程并抛异常。

而MessageRecordTask则是定时任务，内部的reconciliationTask方法每分钟都会被线程池开启线程执行。内部会调用messageRecordService.executeReconciliationTask();方法。

在messageRecordService.executeReconciliationTask();方法内部的执行逻辑是这样的：

1. 遍历所有的MessageType，对每一个MessageType执行同样的对账操作
2. 通过exceptionMessageHandlerContext拿到对应MessageType的实现类，并查询该MessageType下所有未对账，以及对账失败的消息发送记录
3. 并通过这些消息发送记录的消息id，查询到对应的消息消费记录，并存放在Map中，key为消息id，value为对应的消息消费记录
4. 遍历每一个消息发送记录，获取与之对应的消息消费记录，如果不存在消费记录/未消费/消费失败，则将exceptionMessageHandlerContext.getExceptionMessageHandler(messageType).handle(messageProducerRecord);包装成ReconciliationTask，并调用reconciliationTaskQueue.putTask(reconciliationTask);存放到对账消息队列中执行
5. 否则，如果消息发送成功且消费成功，则设置为对账成功

> 说了这么多总结一下就是：
>
> 1. 每个消息类型，如果需要使用消息对账，则实现ExceptionMessageHandler接口
> 2. 然后reconciliationTask定时任务每分钟会调用一次messageRecordService.executeReconciliationTask();方法
> 3. 方法内部会遍历每个MessageType并对每个MessageType调用ExceptionMessageHandler接口的noReconciliationMessageProducerRecordList方法获取该MessageType下未对账的消息发送记录
> 4. 然后获取所有对应的消息消费记录
> 5. 遍历消息发送记录集合，如果消费记录不存在/未消费/消费失败，则将ExceptionMessageHandler.handle()方法包装成对账任务ReconciliationTask，并存放到ReconciliationTaskQueue的阻塞队列中，ReconciliationTaskQueue会有线程负责监听队列，取出任务并开启新线程执行任务。在ReconciliationTaskQueue中只允许失败重试3次，如果三次都未成功则抛出错误日志
> 6. 否则，如果消息发送和消费均成功，则对账成功

通过配置KafkaTopic中的topic，以及ProducerConfig里的相关配置，利用ApiDataMessageSend完成kafka向指定topic发送消息。

主要用于发送那些触发了规则的接口信息。其对应的消费者在定制化服务模块，通过读取消息将对应的apidata存储到数据库当中。

#### 限流与校验部分工具类

Token解析没什么好讲的，就是检查token是否有效，然后再从token里获取用户信息等

其次就是限流器，使用Semaphore实现，通过创建RateLimitProperties配置限流开关和信号量的数目。在RateLimiter里提供限流工具类，请求只有获取到信号量才能进入过滤器执行业务，否则就抛出操作频繁

然后就是api限流部分，基于customize服务里的Rule和DepthRule

* 首先ApiRestrictCacheOperate用于加载Lua脚本，并提供apiRuleOperate执行Lua脚本
* 其次ApiRestrictData则是接口请求记录的实体对象，包含规则触发结果，规则触发的类型，api调用次数，规则阈值，错误信息的规则索引。
* ApiRestrictService则是最后方法的集成。

在ApiRestrictService中核心方法是apiRestrict，判断用户对某个api的调用是否触发规则，并记录用户的调用次数等。他还包含这些子方法：

* checkApiRestrict，用于查询请求的url是否在api限流的名单里，是则返回true，不是则返回false
* getRuleParameter，用于获取普通规则的一些参数，需要传入三个参数：apiRuleType，通用Key，以及具体规则RuleVo。核心逻辑就是将这几个参数传入JSONObject中，并基于这三个参数，构造出其余关键的参数对
* getDepthRuleParameter，用于获取深度规则的参数，需要传入已构造好的普通规则的JSONObject(原因是深度规则基于普通规则)，通用key以及所有的深度规则。将深度规则根据窗口的开始时间排序，然后遍历每一个深度规则，将他的参数填充到一个新的JSONObject中，并最后添加到一个List<JSONObject>中，一起添加进普通Rule的JSONObject中

核心的apiRestrict方法解析：

1. 判断请求的url是否是需要限制的，如果不是整个apiRestrict方法直接返回，否则继续

2. 通过请求，获取到ip，最后与userId以及请求路径一起拼成形如ip_userId_url的CommonKey

3. 从Redis中获取Rule和DepthRule。如果Rule不为空则apiRuleType标记为RULE(该参数初始为NO_RULE)，如果深度规则和普通规则都不为空则更新为DEPTH_RULE（原因在于深度规则是基于普通规则的，普通规则必须存在深度规则才会生效，深度规则就是统计时间窗口内触发了几次普通规则）。

4. 只要apiRuleType是RULE或者是DepthRule（只要不是NO_RULE），那么就调用getRuleParameter获取普通规则参数（主要参数就是apiRuleType，ruleKey(commonKey前加上rule_api_limit)，statTime统计时间，threshold阈值，effectiveTime惩罚时间，ruleLimitKey(普通规则限制key)，zSetRuleStatKey（为深度规则准备））

5. 如果apiRuleType是深度规则，则在此基础上新添加深度规则的参数（因为深度规则基于普通规则，深度规则的处罚流程就是得先触发普通规则，所有深度规则有普通规则的所有参数，因为普通规则的触发流程也得有），比如：depthRuleSize深度规则数量，currentTime当前时间，用户判断是否进入深度规则的窗口，以及List<JSONObject> depthRules，每个JSONObject都包含每个深度规则的参数：statTime深度规则统计时间，threshold深度规则阈值，effectiveTime深度规则惩罚时间，depthRuleLimit深度规则限制key，startTimeWindowTimestamp开始窗口，endTimeWindowTimestamp结束窗口

   > 这里需要补充的是：commonKey，ruleKey，ruleLimitKey，zSetRuleStatKey，DepthRuleLimitKey之间的关系。
   >
   > 1. 首先commonKey就是通用key，规则限制里所有的key都需要靠它拼接，因为他包含用户表示，他的形态如：ip_userId_url表达的意思是userId在这个ip访问里url，能唯一确定一个用户
   > 2. ruleKey是规则限流的key，这里的rule并不是特指普通规则，它是由rule_api_limit + commonKey拼接的，用于记录userId的用户在当前ip访问此接口的一个次数
   > 3. 当ruleKey的次数达到普通规则阈值时，则会生成ruleLimitKey，后续如果这个key存在则会直接不允许访问。
   > 4. 当触发普通规则时，会向ZSET中插入一条触发内容，此时插入的key就是zSetRuleStatKey
   > 5. 当zSetRuleStatKey里的信息超过深度规则的阈值时，则会创建更严格的DepthRuleLimitKey，后续如果DepthRuleLimitKey存在则也会禁止访问，且优先级更高。

   Lua脚本执行逻辑：lua脚本的执行流程；

   1. 先通过rule_key，自增调用次数(rule_key由rule_api_limit和commonKey拼接而成)。如果自增后的count等于1，说明是第一次调用，那么会为rule_key设置过期时间，过期的时间为普通规则的统计时间。
   2. 如果次数超过了普通规则统计时间内的调用阈值，如果rule_limit_key不存在的话则会创建rule_limit_key，并设置过期时间为普通规则的惩罚时间，并将触发规则的类型设置为普通规则。同时将当前时间与调用次数拼接构建z_set_member，以传入的z_set_key为key，当前时间为score，z_set_member为value，插入到zset中。并将trigger_result设置为1
   3. 如果以及存在rule_limit_key，则会将trigger_result设置为1，代表触发了规则
   4. 如果存在深度规则，会逐一遍历深度规则并判断当前时间是否大于深度规则的窗口时间，如果大于则会将zset里score在窗口时间之前的这些无用记录都删除
   5. 再判断当前时间是否处在时间窗口内，如果是的话暂时初始化z_set_min_score为窗口开始时间，z_set_max_score为当前时间。
   6. 如果当前z_set_max_score与z_set_min_score的差值大于了深度规则的统计时间，则会将z_set_min_score初始化为current_Time - (depth_rule_stat_time * 1000)
   7. 统计zset中score处在z_set_min_score和z_set_max_score之间的记录数量，如果超过了阈值，则会触发深度规则的保护，如果不存在depth_rule_limit_key的话会先创建，并设置过期时间为对应深度规则的惩罚时间。并将trigger_result设置为1，trigger_call_Stat设置为2，message_index设置为当前深度规则遍历的index。然后返回这些信息
   8. 如果说depth_rule_limit_key已经存在了则修改trigger_result = 1为1，记录message_index的index，并返回相关信息。

#### Filter过滤器

1. 网关全局过滤器入口

   - 过滤器实现了 `GlobalFilter` 和 `Ordered`，优先级为 `-2`。

   - 每个请求都会进入 `filter(ServerWebExchange exchange, GatewayFilterChain chain)` 方法。

   >  **作用**：决定请求是否通过限流，初始化一些公共参数。

2. 限流处理（信号量 RateLimiter）

   1. 检查限流开关 `rateLimiterProperty.getRateSwitch()` 是否开启。

   2. 如果开启：

      * 调用 `rateLimiter.acquire()` 尝试获取一个许可：

        - 信号量构造时按每秒最大请求数 `maxPermitsPerSecond` 初始化。

        - 如果当前许可用完，抛出异常 `DaMaiFrameException`，提示请求过于频繁。

      * 请求处理完毕后调用 `rateLimiter.release()` 释放信号量。

   3. 如果限流未开启，则直接调用 `doFilter()` 继续处理。

   > **作用**：全局粗粒度的请求速率控制，避免瞬时请求量过大。

3. 请求预处理（doFilter 方法）

   3.1 获取请求头

   - 获取以下信息：
     - `TRACE_ID`：链路追踪 ID，如果没有则生成一个 `UidGenerator` 唯一 ID。
     - 灰度参数 `GRAY_PARAMETER`。
     - `NO_VERIFY`：标识是否跳过验签。
   - 将以上信息存入：
     - **MDC**（用于日志追踪）。
     - **BaseParameterHolder**（线程上下文共享）。
     - 请求头 `headMap`，后续可能透传给微服务。

   3.2 判断请求类型

   - 判断 `Content-Type` 是否为 `application/json`：
     - **是 JSON**：调用 `readBody()` 方法，读取并修改请求体。
     - **不是 JSON**：直接执行 `doExecute()`，处理请求头和业务参数。

4. JSON 请求体处理（readBody 方法）

   1. 使用 `ServerRequest` 读取请求体（`Mono<String>`）。
   2. 对请求体调用 `execute()`：
      - **执行业务逻辑验证**。
      - 对请求体中的加密字段进行解密（如果 `encrypt=V2`）。
      - 验证 RSA 签名。
      - 检查 Token 是否存在、是否有效。
      - 根据 URL 判断是否必须有 `userId`。
   3. 将处理后的请求体和相关参数存入 `RequestTemporaryWrapper`。
   4. 使用 `BodyInserters` 将修改后的请求体写回请求中。
   5. 最终返回给下一过滤器或下游服务。

   > **作用**：保证请求体安全、合法，并统一将必要参数透传。

5. 非 JSON 请求体处理（doExecute 方法）

   1. 将原始请求体解析为 Map。

   2. 检查是否需要进行签名验证（`NO_VERIFY` 或全局配置）。

   3. 获取渠道信息 `channelDataVo`，验证 RSA 签名。

   4. 检查 Token：

      - 如果需要但缺失，抛出异常。

      - 如果存在，调用 `tokenService` 获取用户信息。

   5. 处理业务参数：

      - 对 `BUSINESS_BODY` 解密。

      - 将 `userId`、`code` 等透传。

   6. 调用 `apiRestrictService.apiRestrict(userId,url,request)`：
      - **作用**：调用 API 限流/防刷逻辑（可能就是你 Lua 脚本对应的规则）。

6. 请求头装饰（decorateHead 方法）

   - 使用 `ServerHttpRequestDecorator` 对请求头进行增强：
     1. 将处理后的业务参数透传到请求头。
     2. 添加或覆盖全局参数，如 `TRACE_ID`、灰度参数。
     3. 处理请求体长度：
        - 有 `Content-Length` 用原值。
        - 没有则使用 `Transfer-Encoding: chunked`。
   - 返回装饰后的请求，传给后续微服务。

   > **作用**：保证微服务能够拿到必要的上下文参数。

7. 辅助判断方法

   **skipCheckToken(url)**：判断 URL 是否跳过 Token 验证。

   **skipCheckParameter(url)**：判断 URL 是否跳过参数校验。

   **checkParameter(originalBody, noVerify)**：判断是否需要进行参数检查。

   **checkNeedUserId(url)**：判断 URL 是否必须携带 `userId`

8. 总体请求处理流程总结

   1. **进入全局过滤器** → 检查限流。

   2. **限流通过** → 预处理请求头和链路信息。

   3. **判断请求体类型**：

      - JSON → `readBody()` → 解密、验签、Token 验证、API 限制。

      - 非 JSON → `doExecute()` → 验签、Token 验证、API 限制。

   4. **API 限制调用** → `apiRestrictService.apiRestrict()`。

   5. **装饰请求头** → 将必要参数透传给下游服务。

   6. **继续过滤器链** → 下游微服务处理。

### damai-user-service

用户板块的技术相对比较简单，主要是提供用户相关的查询功能，以及购票人这个属性。为后续的订单以及节目模块服务。

#### 布隆过滤器缓存用户手机号，在业务层预防缓存穿透

UserBloomFilterInitData用于缓存用户手机号，来预防一定的缓存穿透的问题。UserBloomFilterInitData继承了之前设计的初始化任务管理的抽象类AbstractApplicationPostConstructHandler，实现了executeInit方法，其实就是将用户的手机号添加到布隆过滤器中

#### 组合模式处理用户注册前的一系列验证逻辑

这里利用组合模式，依次执行UserRegisterVerifyCaptcha，UserRegisterCountCheckHandler，UserExistCheckHandler。

分别的职责就是：

* UserRejisterVerifyCaptcha负责验证提交的两次密码是否一致，并判断是否需要验证码，如果需要的话判断验证码是否通过
* UserRegisterCountCheckHandler配合RequestCounter判断，注册请求是否达到阈值，若达到阈值这拒绝（同一台JVM里1s内是否有超过1000个用户请求注册）
* UserExistCheckHandler判断当前用户注册手机是否存在，存在则抛异常

#### Lua脚本统一限制分布式场景下的注册频率

Lua脚本的整体流程为：

1. 判断验证码开关，如果是一直需要验证码，则直接返回true
2. 获取当前count计数和上次重置的时间，如果key不存在则均返回0
3. 验证当前时间和上次重置时间的差值是否超过时间窗口的长度（1s），超过则将count重置为0，重置时间戳改为当前时间，并重新设置key和过期时间。否则就count + 1
4. 判断count是否大于阈值，如果大于，则重置count于重置时间戳，设置验证码标识为yes，并返回true
5. 如果未超过限制，则更新count，并将验证码标识设置为no，并返回flase

> 总结就是，判断用户注册次数在统计时间内是否超过阈值，如果超过了，则需要验证码，若未超过则不影响。如果开启了全局验证码，则无论是否超过阈值都要进行验证码校验。

结合前面的组合模式所以整个流程就是：

1. 用户进行注册操作
2. 用户填写注册表单数据
3. 点击注册按钮
4. 判断是否需要验证码（Lua脚本）
5. 如果不需要则直接跳至执行用户注册，若需要则获取验证码则获取验证码
6. 执行用户注册
7. 执行用户注册的验证逻辑
   * 验证两次密码是否一致，进行后端二次验证
   * 验证用户的注册请求次数是否超过阈值
   * 检查注册的用户是否已经存在
8. 添加用户

#### 购票人部分

该部分其实并没有想象中的那么简单，其背后的思考也是值得推敲。

- 首先在生成订单时，涉及到调用购票人列表接口，给用户展示购票人列表信息
- 在生成订单接口中，也要查询购票人列表信息，验证传入的购票人信息是否正确

需要注意这两次操作都是要从数据库中查询，购票人的数据库表是用的user_id作为分库分表的分片键，所以不会有读扩撒导致全路由的问题。

但仅仅是将购票人数据进行分库分表是不够的

查询购票人信息的操作是在生成订单过程中，也就是**抢票业务**。抢票业务可是大麦项目的核心高并发业务，当大量的用户来抢票的话，这个购票人列表的查询可是对数据库的压力很大的，不夸张的说，当请求到达了一定量级了，数据库超时现象很快就能爆出来。

既然数据库的压力这么大，解决的最好方法还是使用缓存，并且购票人修改的概率很低，所以缓存和数据库一致性的问题也很好解决。

但目前的问题是**添加缓存的时机**，有的人会想那在生成订单过程中，把购票人信息放入缓存不就可以了吗，这么做其实不可行，首先从业务入手，大麦网的高并发特点是用户会一瞬间抢购同一场演唱会，抢不到就抢不到了。下一次演唱会的抢购就不知道什么时候了。也就说**用户的购票人信息短时间内是不会复用的**，之所以放入缓存，是为了解决这一瞬间数据库的压力。也就是说放入缓存的时机要在生成订单前

还有另一个问题，**需要将所有用户的购票人信息放进缓存中吗**，如果都这么放入了，那缓存的内存压力会非常的大

为了解决这两个问题，我们再从业务入手，首先来说并不是所有的演唱会都是热门需要去抢购的，对于二三线的明星演唱会来说，一瞬间的并发量并没有很高，只有超一线的明星，例如周杰伦、林俊杰这种明星的演唱会并发量确实会很高

基于这个业务特点，我们就可以在生成订单之前，也就是查看演唱会详情时，如果这场演唱会是热门的话，将购票人添加到缓存

到这里，还有个问题解决，就是查看演唱会详情是不需要登录的，有的用户就是浏览浏览，点击进去看看演唱会详情，虽然查看了，但并不一定会抢票。那么通过什么办法能知道用户是为了想抢票而去点进看演唱会详情的呢？

答案就是 **登录**。其实登录的用户也不一定是要抢票的，但是**不登录的用户一定不是抢票的**

所以基于以上的考虑，最终确定**是热门的演唱会，并且用户是登录的状态下，才将用户购票人放入缓存中**

方案确定了，我们来分析流程：

- 首先是用户在登陆状态下来查看热门的演唱会详情，这时将用户购票人列表放入缓存中
- 但还有另一种情况，就是用户在没有登录的状态下查看热门的演唱会详情，这时不把用户购票人列表放入缓存

如果这时，用户想要抢票了，就必须要登录，然后再去调用演唱会详情，接着会跳转到生成订单或者选座位的页面，而在这个**再去调用演唱会详情的过程中，就还可以将购票人列表放入缓存**

> 还有个问题就是，如果刚上了一个热门节目，然后用户都点进来查看节目详情，此时按照逻辑应该查询购票人信息然后放入缓存，但此时比如说是周杰伦这类超级明星的号召力，导致大量用户拥入详情页，进而使得数据库压力激增该如何处理？
>
> 其实不提出这个问题也应该意识到，添加缓存不应该跟查看详情耦合。就必须定时任何和异步

#### 私密信息的加密存储和脱敏展示

在加密存储中，主要通过使用ShardingSphere的加密配置，在分库分表时对特定数据列使用特定的加密算法存储。

脱敏展示主要是在vo类中使用如Hutool的StrUtil.hide或者DesensitizedUtil.idCardNum来对某些关键部分进行星号处理。

```java
public String getRelName() {
        if (StringUtil.isNotEmpty(relName)) {
            return StrUtil.hide(relName, 0, 1);
        }
        return relName;
    }
    
public String getIdNumber() {
        if (StringUtil.isNotEmpty(idNumber)) {
            return DesensitizedUtil.idCardNum(idNumber, 4, 4);
        }else {
            return idNumber;
        }
    }
public String getIdNumber() {
        if (StringUtil.isNotEmpty(idNumber)) {
            return DesensitizedUtil.idCardNum(idNumber, 4, 4);
        }else {
            return idNumber;
        }
    }
```

> StrUtil.hide是纯字符串拼接，而DesensitizedUtil则是把“字段含义”封装成方法名，可读性更强，但他只能覆盖**内置字段类型**，灵活性不及StrUtil.hide

* 收到支付包回调后如何更新数据？如果再延迟订单取消的最后一秒用户付款了，如何处理？
* 为什么要payCheck？支付宝回调缓慢如何处理？

#### 生成订单

创建订单无非就是生成对应的订单号很简单。他有两个主要入口，一个是create方法一个是createByMq。后者主要是kafka异步创建订单时使用，区别就是传入的参数createByMq对了一个identifierId这里不做介绍。最终两个入口都会调用doCreate方法。

doCreate的逻辑：

1. 根据订单编号orderNumber查询order，如果订单编号对应的订单已存在直接抛异常
2. 遍历传入的用于创建购票人相关订单和记录的List
3. 根据传入的购票人信息，为每一个购票人创建OrderTicketUser和OrderTicketUserRecord（前者是购票人的订单，后者相当于市操作日志，记录这条订单对应的操作是什么，比如说下单就是RecordType.REDUCE也就是减少库存）
4. 最后遍历完后，插入主订单Order，批量插入购票人订单和购票人订单记录。为了方便后续每个节目下的对账任务，还需要插入OrderProgram节目订单，并标识为未处理
5. 最后在redis中自增用户在该program下的下单数（有多少个购票人就增加多少

#### 支付

订单服务里的支付逻辑很简单，因为是远程调用的支付服务的方法。

1. 验证订单是否存在，不存在抛异常
2. 验证订单状态是否是已取消，如果是则抛异常
3. 验证订单状态是否是已支付，如果是则抛异常
4. 验证订单状态是否是已退款，如果是则抛异常
5. 验证传入的金额与数据库中的金额是否匹配，不匹配则抛异常
6. 构建支付调用的参数
7. 调用支付服务，如果响应码不是成功的响应码则抛异常
8. 若以上判断都成功则返回响应体

> 需要注意，这里调用支付服务得到的支付平台的响应其实是支付平台确认收到订单，并生成支付页面。看到的场景就是跳转到了对应支付平台的付款页面。
>
> 然后用户在这个页面支付后，支付平台就会展示支付成功/支付失败的页面。页面由支付平台渲染，并且支持传入return_url支持跳转回项目的前端。
>
> 然后支付平台再异步调用回调接口，传递支付结果等信息，服务端需要对回调的信息验签等

#### 支付回调

支付回调由支付平台在支付后主动调用项目的接口

1. 拿到回调的请求体
2. 从请求体中拿到订单号，并执行try-catch代码块
3. 如果订单号对应的订单为空，直接抛异常
4. 如果数据库的订单的状态为CANSEL，这说明用户很可能是延迟订单的最后一秒点击支付，导致数据库里订单取消了，但请求已经发到支付平台了。那么此时的策略就是调用cancel退款接口，并返回退款的响应
5. 以上情况都不是的话，就执行支付服务的回调逻辑（首先是验证签名，通过订单号查询支付账单并判断：账单是否存在，账单状态是否正确，回调传入的参数是否匹配）
6. 最后根据支付服务回调接口的响应来做进一步操作，如果没成功则抛异常，如果成功了则更新相关记录的状态

#### 主动确认支付状态

因为支付回调是异步的，如果回调慢就会导致用户扣钱了但是订单显示未支付。

> payCheck的这个调用发生在用户完成支付宝支付并被重定向到支付成功页面后，用于确认订单的最终支付状态。

为此需要设计一个主动确定支付状态的逻辑，也就是payCheck

1. 验证订单是否存在，不存在则抛异常
2. 如果订单已取消，则说明用户在延迟订单取消的最后一秒支付了，处理逻辑就是调用退款的逻辑
3. 否则调用支付服务的tradeCheck方法，查询支付状态
4. 如果响应未成功则抛异常
5. 从响应中获取响应体，响应体才是查询支付状态的响应.
6. 如果查询支付状态失败，则抛异常
7. 否则判断查询的支付状态和数据库订单状态的关系（必须得是两者不相同），如果查询的支付状态为已支付则执行updateOrderRelatedData方法更新数据为支付，若状态为已取消则也调用updateOrderRelatedData方法，不过更新状态为已取消

> **`alipayNotify`**：
>
> - 依赖支付宝的重试机制
> - 如果处理失败，支付宝会重复发送回调，直到收到`success`响应
>
> **`payCheck`**：
>
> - 作为异步回调的补充，通常用定时任务或手动触发的补偿机制
> - 可以定期检查那些状态未同步的订单

> paycheck会调用支付服务里的traceCheck方法，该方法查询的是支付宝那边的支付状态，如果查询出来的状态和项目中的支付账单PayBill的状态不同还会顺便修改。然后查询成功后，如果项目中订单状态和支付账单状态不一致就会去更新对应的订单状态
>
> alipayNotify在执行过程中，会判断PayBill的状态以及订单的状态，所以不会出现paycheck修改了一次alipayNotify又修改了一次。

#### 更新订单相关数据

`updateOrderRelatedData`该方法主要就是更新相关数据，主要用在取消订单，支付宝回调接口以及支付检查中。负责将未支付的订单按照具体的目标操作状态如已支付/已取消，更新至对应的数据。

1. 首先更新数据只允许两种订单操作状态：支付和取消。支付做的事情就是把相关票档下的座位等信息变成已销售，并且更新相关订单的状态。取消的话则需要做数据恢复。
2. 检查订单状态，只处理未支付订单，其余状态的订单直接报错
3. 根据订单号拿到该订单下的购票人集合。分别创建对应的更新对象updateOrder和updateTicketUserOrder，并设置状态为目标状态（已支付/取消）
4. 判断目标状态，如果是已支付则操作记录的类型就是改变状态（这里指的是改变座位的状态：锁定 -> 已售出），如果是已取消则记录类型为增加余票（其实也是另一种形式的改变状态，只是为了区分这是已取消操作的行为：已锁定 -> 空闲）
5. 更新订单和购票人订单，并记录更新的订单数和购票人订单数
6. 遍历购票人订单集合，添加购票人订单的记录流水（第四点提到的操作记录类型），并将购票人id和其对应的座位id封装为一个对象，添加到一个集合中
7. 如果是取消的操作还要做一件事情，就是把用户在该节目下的订单数量减去取消的购票人数
8. 最后将数据又拼成两个Map，一个Map存储节目票档id下的购票人订单集合，另一个Map存储节目票档id下的座位id即可
9. 最后将上述整理的参数传入实际的更新方法中`updateProgramRelatedDataResolution`

`updateProgramRelatedDataResolution`该方法整体流程就是根据传入的相关参数，根据目标状态，封装好参数以及redis的key，并传入对应lua的执行脚本中，在lua脚本中执行具体的数据更新逻辑。

他的内容比较难，会讲的跟细一点：

##### 一、updateProgramRelatedDataResolution 是在干嘛

>
> **把“订单支付 / 取消”对节目库存、座位状态、座位锁、票档余量、操作流水的影响，**

**5.1 校验座位没被卖过**

```java

```

原因：

- Redis 才是真实库存
- 节目服务数据可能滞后
- **只做防重复售卖兜底**

**5.2 更新座位为 SOLD**

```java
update seat
set sell_status = SOLD
where program_id = ?

```

**没有取消逻辑**

```java

- 扣库存
- 校验更新条数
- 不一致直接回滚

**V1–V3 小结**

- Redis 已经处理真实库存
- 节目服务只是 **最终同步数据**
- 只关心“卖出”

> 对于V1-V3版本：
>
> - 当订单状态为 **PAY** 时，不仅执行Lua脚本更新Redis缓存，还会发送延迟消息去更新数据库
> - 当订单状态为 **CANCEL** 时，只执行Lua脚本更新Redis缓存，但不发送延迟消息更新数据库
>
> 这是因为V1-V3版本的设计中，只有**支付**操作才会触发对**数据库的最终状态更新**，而**取消**操作只需要**回滚Redis中的临时状态**即可。

###### 🔹 V4 执行流程（重点）

------

###### 6️⃣ 校验座位必须是 LOCK

> **只能操作“我亲手锁定的座位”**

- SOLD → NO_SOLD（绕过订单）

- 校验执行条数

- 并发冲突

#### 延迟队列消费

> 其实还有一个发送，这里就简单说一下，这个延迟订单发送主要就是给节目服务发送一条消息，让节目服务把相关的数据库信息给改了。就这么简单

这里的延迟订单消费处理的就是延迟订单取消，在节目服务下单后，若两分钟内用户未支付，就会出发延迟订单取消的任务。

项目中为了确保消息的可靠，添加了消息对账的功能。（联合着节目服务的延迟订单发送一起分析）

1. 生成消息链路id和消息id，并和节目id和订单number一起封装为DelayOrderCancelMessageModule。并将DelayOrderCancelMessageModule压缩为一个JSON格式的messageContent
2. 构建消息发送对象InsertMessageProducerRecordDto，设置消息类型，消息链路id，消息id，消息的业务id（在这里就是节目id），消息主题，消息内容等。然后调用远程服务插入消息发送记录。如果得到的响应是失败的，则直接记录日志后就反悔
3. 拿到响应得到的消息发送记录MessageProducerRecordVo。尝试向延迟队列发送任务，如果发送成功则同时修改MessageProducerRecordVo的发送状态为成功，反之发送失败

后续就是消费的逻辑，其实和发送时差不多

1. 将收到的消息体解析为DelayOrderCancelMessageModule，并从中拿到相关的消息id，消息链路id，节目id，订单number等
2. 然后根据消息id查询对应的消费记录（只允许两种状态：1. 没有消费记录，代表初次消费 2. 有消费记录，但消费状态是失败，代表正在重试），如果判断为其他状态，或者在查询时出现问题直接记录日志并返回
3. 那如果不存在消费记录的话，那就插入一条，同时获取两个关键参数：消费id和消费次数。如果存在消费记录则直接获取这两个参数，并让消费次数+1
4. 然后尝试消费任务，也就是取消订单。同样如果是取消成功，则修改消费状态为成功，失败则消费状态为失败

#### 对账以及补偿

> 这部分是订单服务除正常业务外，最重要的部分

首先是对账任务的入口，也就是一个定时任务reconciliationTask，该定时任务每三分钟执行一次

1. 调用节目服务的select方法，查询三分钟前所有的未处理的对账记录
2. 从对账记录中拿到对应的节目id和创建时间
3. 对查询到的每个节目id调用对账服务的reconciliationTask进行对账
4. 对账完成后修改这些对账记录的状态

现在主要讲一下OrderTaskService里的这个reconciliationTask方法

首先他其实就分五个大的步骤

1. 先从redis中拿到当前programId下的所有ProgramRecord（key为recordType_identifierId_userId，value为ProgramRecord的JSON字符串）
2. 查询当前programId下所有**未处理**的OrderProgram，这个OrderProgram其实就是当前节目下的所有订单，在这里只查了**未对账**的节目订单
3. 拿到前两步查到的OrderPrograms和ProgramRecords，一个代表数据库里的记录，一个代表redis里的记录。然后还是对比，找出Redis中缺少的记录。这部分呢就是需要补偿的。得到的是一个needToRedisRecordMap（key为identifierId_userId， value为ProgramRecord列表(按reduce/changeStatus/increase排序)）
4. 进行补偿，为当前programId的节目将needToRedisRecordMap补偿进redis的缓存记录里
5. 最后构建返回结果ReconciliationTaskData，包含programId和添加的redis记录即Map<String, ProgramRecord> addedRecords

前两步没什么好讲，就是从数据库和redis中直接查数据拿到各自的记录。第三步和第四步是主要逻辑。

##### findNeedCompensationRecords寻找需要补偿的记录

   4. 将这些数据回写到programRecord中
   5. 还原完后需要reverse回来
3. 随后将补偿后完整的needToRedisRecordMap，将他的数据转移为Map<String, ProgramRecord> completeRedisCordMap，（key为该类型记录的key即recordType_identifierId_userId，value为对应的记录
4. 删除Redis中相关票档的座位和余票缓存
5. 调用programRecordHandler.add方法将completeRedisCordMap构建好的redis记录，添加到redis中

在programRecordHandler.add方法中

1. 记录重试次数，最多允许重试5次
2. 将redis缺少的记录key和redis已有的记录key都存放到keyList中
3. 更新数据库，programId下userId的identifierId则次订单操作为对账成功
4. 删除redis的旧记录
5. 将记录添加到对账后的Hash里，并添加上缺失的记录
6. 若期间操作有异常抛出，则记录日志，判断重试次数，并重试

# 常见问题的处理

## 对账任务里，Redis有记录Mysql没有会怎么办？

实际以mysql为准，redis扣减成功，是获取了下单的机会，表示为锁单。进入付款页面。付款之后会扣减mysql的数据。此时如果扣减成功说明mysql有数据，你成功买到了。但如果出现redis记录比mysql多，那redis里的库存是会更少的，导致redis库存没了，但实际mysql库存还有。

这就是典型的少买问题。结合业务场景（演唱会门票贩卖系统），少买完完全全就不算什么大问题，因为卖演唱会门票是提前1-2个月卖，这段期间完全能处理少卖的问题。比如卖几次，或者将一定库存打包卖给票务。

所以系统里并没有处理少卖的问题

> 这种情况通常说明订单流程失败，常见的处理就是释放这部分的redis库存，或标记为“无效扣减”

## 对账任务里，Redis没有记录Mysql有会怎么办？

总的来说就是找到mysql里的记录和redis里的记录。对比找到所有redis没有的记录（因为以mysql为准，redis有记录代表）然后将这部分没有的记录补上去。

## Kafka消息丢失怎么办？

### kafka层面

**生产者方面**

1. 配置acks=all，这个是控制Broker 什么时候返回“发送成功”。
   * 0代表不等 broker 确认，直接返回（可能丢消息）
   * 1代表leader 写成功就返回
   * all代表所有 ISR 副本同步成功才返回（最可靠）
2. 配置retries参数
   * 网络瞬断会自动重发
3. 开启幂等配置enable.idempotence=true
   * 开启后Kafka 会：
     - 给 producer 分配 PID
     - 每条消息带 sequence number序列号
     - 自动去重

**消费者方面**

1. 配置消费者组group-id。同一个 group 里的消费者：
   * 会分摊 partition
   * 保证一个 partition 只被一个消费者消费
2. 关闭自动提交offset， enable-auto-commit=false
   * 只在消费成功后手动提交。消费还没处理完就提交
   * 宕机导致丢处理
3. auto-offset-reset当没有 offset 时，从哪里开始消费。(offset其实就是消费者消费到哪个消息了，broker只需要将offset后的消息给消费者即可)
   * earliest，从最早开始
   * latest，从最新开始

**broker相关的配置**

1. replication.factor，配置每个partition的副本数。通常>=3
2. min.insync.replicas，配置最少同步的副本数，配合acks=all。比如配置为2代表：至少需要两个副本同步后才返回ack
3. unclean.leader.election.enable，表示是否允许从不同步副本选 leader。配置为false，避免丢数据

### 业务层面

设计消息对账系统。相关的有两张表：MessageSendRecord消息发送记录，MessageConsumerRecord消息消费记录。

首先通过消息包装，将kafka的kafkaTemplate.send方法包装成

`public void sendMessage(String message, SuccessCallback<SendResult<String, String>> successCallback, FailureCallback failureCallback)`

SuccessCallback和FailureCallback是自定义的函数式接口，允许传入lambda表达式或自定义方法来实现当kafka消息发送成功或失败之后的一些自定义逻辑。

那么在kafka消息发送之前会插入MessageSendRecord，然后根据kafkaTemplate.send方法返回的completableFuture调用whenComplete方法监听消息是否成功抵达**broker**（注意这里是抵达broker而不是消费者，因为kafka采用的是pull模式，消费者主动pull任务执行，而不是传统的将消息直接push给消费者。）

```java
if (Objects.isNull(ex)) {
    successCallback.onSuccess(result);
} else {
    failureCallback.onFailure(ex);
}
```

这里的判断很简单，只要future正常抵达，ex就是null。反之就有异常信息说明发送出错了。

这里的话发送成功就在successCallback.onSuccess里讲发送记录的发送状态修改为发送成功。

发送失败的话，由于配置了kafka的retries参数，大部分异常kafka都能自己帮我们去重试，所以这里可以考虑只修改发送状态为失败，然后记录日志。不再继续重试，交给后续对账重试，或人工介入

> 这里面试官很可能会来一句：既然这里只能知道消息抵达了broker，但是这并不意味着消息最终会被消费者消费吧。那这一层你又是如何处理的。
>
> 其实回复也很简单：whenComplete只能判断到消息是否成功写入 broker，并不能判断消费者已经消费这没错。但kafka的设置就是让生产者和消费者解耦，只要消息能够成功抵达broker并且partition的配置正确合理，那么kafka就能保证消息不丢失。至于消费者是否能拿到消息，是由 Kafka 的副本机制和消费模型保证的，而不是生产者负责的。只要消息持久化到broker上就一定可以被消费。此外即便真的丢了，消息的发送记录已经存在数据库里了。可以在后台或者说设计一个定时任务，定期扫描一些异常记录，比如创建消息发送记录创建了很久但是一直没有对应的消费记录。然后人工介入去处理。

此外同样的，在消费消息时也会在消费之前会判断当前是否存在消费记录。如果不存在则说明是第一次消费，则插入MessageConsumerRecord。如果已经存在则先判断消费状态，如果是已消费则直接返回成功(这里确保了消费幂等性)。如果消费失败，则记录当前的消费次数（最多重试三次）然后重新消费。消费成功修改消费状态为success，反之失败则修改为fail。

那记录发送状态和消费状态肯定是不够的，需要定期对账，对那些异常的消息做扩展。

所以就会有一个定时的对账任务。会扫描一定时间范围内未对账的MessageSendRecord。然后判断这些消息发送记录的状态，如果是发送就失败了，那就尝试再重新发送。然后将那些发送成功且未对账的MessageSendRecord收集起来。通过他们的id找到对应的MessageConsumerRecord，如果消费记录不存在，或者说消费失败或者是未消费。那就重新消费一次。相反如果发送和消费都成功，那就将他们都标记为对账成功。

> 注意实际的对账任务是分了不同消息类型的。因为这算是一个异常的处理，处理消息消费失败发送失败，消息丢失的一种方案。所以业务层面定义了ExceptionMessageHandler接口开放了三个行为：消息类型，获取该消息类型下没有对账的消息发送记录集合，以及出现故障处理消息的方法
>
> 那么项目启动时呢就会拿到所有ExceptionMessageHandler实现类，然后将他们按照MessageType缓存到一个Map中。对外提供get方法，充当工厂



今天腾讯面试官问到大麦项目一个问题，就是问我kafka的消息如何保证不丢失的。

 我给他说的是：

* kafka生产者方面配置acks=all，也就是所有 ISR 副本同步成功才返回，然后开启幂等配置。
* 消费者层面关闭自动提交offset，只在消费成功后手动提交，不允许消费还没处理完就提交。
* broker层面配置，并且配置最少同步的副本数，确保有这么多副本完成持久化后才ACK。 

然后我把延迟队列那套加在了kafka这里，我说在发送消息时生成一个消息的唯一ID并且插入消息发送记录，消息发送成功就修改状态为成功，有异常就在catch里修改状态为失败。消费时也一样，先判断消费记录是否存在，如果存在说明在重试，判断重试次数，然后再重试，重试成功就修改状态，如果重试了几次都失败就修改为失败。如果不存在消费记录就证明是第一次消费就插入消费记录，然后消费任务。然后后台会有定时任务去判断一个消息的发送和消费是否对得上。

其实无非就是

1. 发送失败
2. 发送成功，但没来得及修改状态服务宕机了（这时有两种情况，因为消息发出去了。因此可能出现消费成功和不成功）
3. 发送成功 消费失败 
4. 发送成功消费成功。 

对于第一个情况，发送就失败了，我业务里包装了kafka的send方法传入一个failCallback，发送失败，会尝试恢复锁定的座位，如果恢复失败也没关系，其实就是少卖，因为扣减库存以redis库存为主，redis扣成功了才能扣mysql，redis这里少了mysql就有余票，只要不是超卖就还好。

对于第二个情况，如果消费成功就修改发送状态，并标记为已对账，如果消费失败或者说没消费记录，就重试，重试都失败了就标记为对账失败，人工处理。

对于第三个状态，就是重试，重试成功，就已对账，失败就对账失败。

对于第四个，直接对账成功。 

同时我还包装了kafka的send方法，加入了两个函数式接口，SuccessCallback和FailureCallback，拿到send方法返回的CompletableFuture，调用whenComplete方法，看是否有异常，有异常就执行FailCallBack，没有就执行SuccessCallBack。这个whenComplete就代表消息抵达了broker并且被持久化了并且producer收到了ACK，其实这里producer的任务就结束了，因为kafka消费消息好像是消费者主动拉取，至于消息是否被消费成功得看消费者这里怎么样。 

然后重点来了，面试官就说“kafka如果在持久化时broker宕机了，或者服务宕机了咋整？消息就直接丢了不要吗”，我说不影响，既然没持久化成功，自然就是发送失败，消息都没发出去，消费者也不用消费，只不过redis的库存会被占用，会出现少卖的问题。那即便broker宕机了，那消息如果消费了就正常走业务，没消费那就是少卖的问题，只要没超卖就是可以接受的。他就沉思了，说“这个broker持久化了，其实并不代表消息就百分百持久化了，或者说最后就能消费到了”，我指出，即便有问题，最多也是少买，可以通过二开三开继续买，或者是卖给票务，尽可能降低损失。面试官就说这也只是另一方面的取舍，但对于你的系统来说设计师有缺陷的。

> 这部分的关键点在于ACK=all 并不等于绝对持久化
>
> 因为Kafka的写入流程是：Producer -> Leader Partition -> 写入Page Cache -> Follower 同步 -> 返回ACK -> OS刷盘。
>
> 那么Kafka 返回 ACK 时，数据很可能只在 PageCache 里。
>
> 并且kafka官方文档也指出“Kafka不能保证在极端崩溃情况下零数据丢失。”
>
> 面试官真正问的其实就是“ACK 已经返回，但数据没真正落盘。”此时broker宕机，那就会导致消息其实并没有持久化到磁盘，但是你的producer确实收到了ACK，并且消息发送记录也标记为发送成功。
>
> 但此时由于没消息，因此消息就不会被消费，也就没有消费记录。对账时发现发送成功但没有消费记录，对账系统会认为是：还没消费或消费失败，会立刻重试。因为此时ACK已返回，那就有两种情况：
>
> 1. 消息发送成功，并且有消息发送记录，发送记录里有消息内容，可以重新发送
> 2. ACK返回后，在修改状态前宕机，也就是发送失败，这是少卖问题
>
> 比较轻量级，更严谨的就是设计本地消息表，先写消息表，然后后台线程或者说其他服务，根据这个表去发送kafka，这样即便broker没持久化成功，消息也持久化在了表中

## 下单时可能出现的一些连环问题：

### 1、Redis扣减库存成功后，宕机了怎么办： 

这里需要针对宕机的时机针对性考虑，因此这里会出现两种宕机情况：

* redis执行完lua脚本扣减成功后还没来得及响应客户端就立刻宕机
* redis在响应了客户端后才宕机

对于第一种情况，也就是扣减完直接宕机：那这里其实由于还未响应就直接宕机，就可能会导致客户端这边连接超时，拖慢响应。但实际业务表型还是等价于没抢到。这种情况应该考虑的应该是redis高可用，以及幂等设计，防止重复扣库存以及重复发送消息等

对于第二个情况，也就是响应了客户端才宕机，那这里由于已经响应了，因此服务端是知道扣减成功了的，会执行后续的发送kafka消息，并消费kafka消息等逻辑。但由于此时redis宕机，如果此时没来得及持久化这部分更新，可能导致redis没有记录下此次库存修改，导致redis库存 > mysql。但这其实并不影响，因为最终是否抢到票判断的是mysql中的库存数量，因此这只会导致少卖的问题。

但情况二还可能发生一个问题就是：redis库存一致大于mysql，导致大量其实无效的请求打入mysql，但这些其实因为redis库存 > mysql才判断为能够购买的，但由于实际mysql库存更少，其实买不了。就会导致mysql处理大量这样的订单。

解决的方案：通过对账方案修复redis和mysql的不一致问题，redis多了就释放，mysql多了就补偿

### 2、lua在Redis执行扣减过程中，宕机了怎么办：

这个其实不懂担心，因为redis保证在执行lua脚本时要么全部执行完，要么都不执行。换句话说redis保证了lua脚本执行的原子性。那lua在执行扣减的过程中宕机了，那当前redis宕机直接就体现为他的内存数据全丢嘛，全丢了业务上的表现和用户没有抢到票是一样的啊，因为我们判断用户是否抢到票还是redis库存扣减 + mysql库存扣减。

并且只有redis扣减成功才会扣减mysql，因此lua都没执行完mysql自然也不会有记录，宕机后redis也不会有你的记录。自然和你没抢到票的表现一致。

这里其实更应该考虑的问题是在高并发场景下频繁发生这种情况，本质是系统抗压能力不够，Redis可能成为瓶颈。

对应的应对方案就是保证redis高可用

* 比如上主从 / 集群
* 控制Lua脚本复杂度，避免执行时间过长
* 限流，避免把Redis打爆

### 3、扣减库存成功后，消息队列MQ宕机了怎么办：

### 4、扣减库存成功后，消息队列MQ的消息丢了怎么办： 

对于这两个问题，解决的方案其实就是设计一个消息记录表，主要分为消息发送和消息消费记录。

在发送kafka消息前先插入消息发送，消费前也先插入消息消费。然后分别在发送成功和消费成功或者在是catch中对消费失败和发送失败的这些情况修改对应记录表的状态。并记录当前消息的内容等标识。

然后通过后台的对账任务来判断某个消息是否消费成功，再根据不同的情况做处理。

* 比如扣减库存后kafka宕机，如果在发送前宕机那其实就是少卖，交给mysql和redis的一致性同步。
* 消息发送成功后kafka宕机，那其实发送成功后已经有对应的消息记录了，可以统计消息记录里的内容做重试或后台记录等操作
* 如果消息发送成功但消息丢了，那由于发送记录表已经记录了消息的具体内容，因此可以尝试重试等
* 如果消息消费过程中kafka宕机了，那由于消费记录表的状态不是消费成功，因此在对账任务中会重新发送尝试执行
* 如果消费成功，但没来得及修改消费状态就宕机，为了避免重复消费，需要保证幂等性，比如插入幂等表，当我的业务里没那么复杂，因为就是下单操作，在消费逻辑里直接判断订单状态，比如支付操作只处理未支付状态的订单，退款只处理已支付的。通过这个状态来维持幂等

### 5、扣减库存成功后，消息队列MQ没有宕机，但消息延迟了怎么办： 

延迟超过一定阈值的消息直接丢弃，并上报给Promethus。原因在于超过一定阈值说明订单的状态变得不可信，用户有可能等不急刷新页面重新下单，也有可能出现消息堆积导致延迟订单取消已经触发并恢复了库存，此时如果还消费就会出现超买

### 6、Kafka消费逻辑里为什么会判断当前时间和订单创建时间的差值是否超过一定阈值（比如5s）？

这个问题的本质就是：业务是强时效性的，但kafka只是“异步排队”

当前业务中下单的整体流程为：

1. 先在redis中扣减redis中的库存
2. 随后构建订单创建的相关参数，比如订单号，节目id，票档id，票数等等
3. 将构建的这些参数发送给kafka（这里如果发送消息出现异常会回滚redis中扣减的库存），kafka消费成功后返回订单号，最后发送订单延迟消息
4. 在kafka的消费逻辑中会判断当前时间和订单创建时间是否超过阈值，如果超过阈值会将订单写入redis的废弃订单中并上报给Promethus。否则就调用订单服务里的创建订单的方法（在这里如果上述过程出现任何异常都会将订单加入redis废弃订单并上报给Promethus）
5. 订单服务中会修改mysql中的库存与座位状态（这一步如果出现问题，则会加入redis废弃订单，但不用上报。原因在于这里会抛异常，会被kafka消费者那里捕获，在那里自然会上报，这里上报会导致重复），随后创建真正的相关的订单表并插入mysql中，最后返回订单号。这样整个下单流程算是结束了

需要注意的是通过redis扣减成功并成功发送消息会进入订单创建的加载页面，但kafka消费成功后才会进入支付页面。

那么问题来了，抢票高峰期大量用户抢票，因此就会有大量创建订单的kafka消息，很有可能出现kafka消费延迟，因为本身消息队列就是异步解耦削峰的。那如果出现消息堆积，导致消息的消费延迟了，并且延迟的时间超过了延迟队列的时间，那这样就会导致该订单本身是失效的但却被错误创建。（在这里即便不这么处理，也只是少卖问题）

因此会判断创建时间是否已经超过了5s，这是项目定义的一个安全时间，超过这个时间，那订单的状态就会很危险，用户很可能等不及刷新页面重新尝试下单了
