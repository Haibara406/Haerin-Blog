### 一个请求的完整调用链可能涉及多个服务，如何能够快速地定位某个接口的性能瓶颈？

答：首先我会按照链路，指标，日志等步骤逐一排查。首先会基于项目中的链路，查看一次请求在各个服务中的耗时分布。比如请求可能会经过：网关 -> 服务A -> 服务B -> redis -> mysql -> 等等。通过查看各个服务的耗时分布，快速定位是服务本身慢？还是远程调用慢？还是数据库操作慢。定位到慢点后，会根据Prometheus或者是grafana等指标监控，来验证瓶颈来源。比如JVM指标，cpu/内存抖动，数据库连接数耗尽等等。最后就会通过服务内部的埋点日志来进一步分析慢的原因。是db查询耗时还是redis调用耗时，还是代码中存在循环或者是复杂的计算，然后精准定位对症下药。总的来说一般我遇到的性能瓶颈的问题可能就是：数据库慢查询（缺少索引），redis缓存失效导致mysql访问量增加，第三方api慢，线程池打满等。然后找到慢点后不仅需要修复慢的代码，更要设置如超市策略以及降级策略，确保服务不会因为个别原因而完全无法使用

### 如何自己实现一个标记复制算法？（核心为如何进行可达性分析以及如何实现标记和复制两个动作）

实现一个标记-复制算法，首先其实有三步。

1. 可达性分析，用于判断一个对象是否是垃圾
2. 标记，将可达性分析能触及的对象标记为存活
3. 复制，准备两个内存区域，from和to。（最开始这些对象都在from区里）将from区里的被标记的对象复制到to区，然后清空from区。最后将to区和from区互换角色。下一次分配内存直接在新的from区分配即可

#### 可达性分析

首先可达性分析就是从GC Roots出发，沿着引用链遍历，能够遍历到的对象都是存活对象。

常见的GC Roots：

1. 比如虚拟机栈中引用的对象，只要方法没结束，里面的被引用的一定不会被回收
2. 类似的还有本地方法栈也就是JNI里引用的对象
3. 还有比如字符串常量池里的对象
4. 方法区中的静态变量引用的对象
5. 被synchronized持有的对象
6. 然后除此之外JVM还会维护一些特殊的对象，比如classLoader，Thread等等

##### 为什么可达性分析一定是从GC Roots开始？

首先，JVM中对垃圾的定义就是从GC Roots不可达的对象，如果不是从GC Roots开始找，根本无法定义什么是“不可达”。再者，垃圾为什么是垃圾？不正是因为没人使用了吗？而Java里没人使用你就代表没有人持有你的引用。而GC Roots是引用链遍历的起点，只有从他开始沿引用链遍历才能知道该对象是否在被引用。

##### 那可达性分析具体该使用哪种遍历方式呢？

首先，对象的引用最终构成的一定不会是数，因为对象可以循环引用，比如A引用B，B引用C，C引用A。

所以对象的引用组成的是图。

对于图而言，常见的遍历方式就两种：DFS和BFS

> 有时候面试官可能会挖坑，说前序，中序，后序应该用哪种。记住：哪种也不是。因为这些是树里才有的遍历方式。
>
> 硬要类比的话，那DFS也是更像前序遍历的那种写法。因为中序和后序遍历的顺序分别是左根右和左右根。那有个问题就是都是得先遍历子节点再去看父节点。而可达性分析是从GCRoots开始的，必须先访问根再访问子节点。这样的话就符合前序遍历的特点：根左右。

实际上DFS和BFS都可以，如果用栈来模拟就是类似前序遍历的DFS，如果用队列来模拟就是BFS（类似于层序遍历）

这里讲解BFS实际的算法大致就是：

1. 初始化，生命一个队列workQueue
2. 将目前所有的GC Roots放入workQueue中
3. 但workQueue不为空的时候，开始循环，从workQueue中取出一个对象，如果对象已经被标记则跳过，未被标记则标记为存活。然后遍历该对象内部的所有引用字段，将这些引用字段也假如workQueue中
4. 但workQueue为空，则遍历完毕，存活的对象就标记得差不多了。
5. 接下来开始复制，准备两个大小相等的内存区域，或者是将原来的内存分为两个半区，from和to
6. 遍历from空间，将标记的对象按顺序复制到to空间，然后更新引用
7. 清空from空间，并且交换from和to空间的角色。

> 至于标记算法如何标记，有这么几种做法：
>
> 1. 在对象头的markword里做标记。
> 2. 可以用一个外部的bitmap来标记，其实就是将堆的内存某个块用一个位来标记。就像我们的内存是一页一页的，堆中的内存可以分成一块一块，而对象就是在一块，或者多块内存上。根据对象所在的地址和堆的起始地址就可以算出对象是在第几块上，然后用一个位图中的第几位在置为 1 ，表明这块地址上的对象被标记了。

### 比如说有一个场景就是假如说这个list里面存的都是int数字，就1234这种。然后假如说给你一个指定的一个数字target，然后把和这个相等的数字全删了，你会怎么做？

1. 使用迭代器

   ```java
   List<Integer> list = new ArrayList<>(Arrays.asList(1, 2, 3, 4, 3, 2, 3, 4));
   int target = 3;
   Iterator<Integer> iterator = list.iterator();
   while(iterator.hasNext()){
     if(iterator.next() == target){
       iterator.remove();
     }
   }
   System.out.println(list); // [1, 2, 4, 2, 4]
   ```

2. 如果是Java 8以上可以用removeIf

   ```java
   List<Integer> list = new ArrayList<>(Arrays.asList(1, 2, 3, 4, 3, 2, 3, 4));
   int target = 3;
   list.removeIf(n -> n == target);
   System.out.println(list); // [1, 2, 4, 2, 4]
   ```

3. 使用普通for循环，但是必须是倒叙，正序可能导致下标错位

   ```java
   List<Integer> list = new ArrayList<>(Arrays.asList(1, 2, 3, 4, 3, 2, 3, 4));
   int target = 3;
   
   for (int i = list.size() - 1; i >= 0; i--) {
       if (list.get(i) == target) {
           list.remove(i);
       }
   }
   
   System.out.println(list); // [1, 2, 4, 2, 4]
   
   ```

### 百度实习面经

Redis的主从同步是怎么实现的

> 第一次连接，会执行全量复制，主节点会在后台生成一份 RDB 快照文件发给从节点，从节点拿到后先清空自己的旧数据，然后加载这份快照。
>
> 在生成和发送快照的这段时间里，主节点不会停止服务，它会把这段时间新收到的写命令，先暂存在一个叫 `Replication Buffer` 的内存缓冲区里。等快照发完了，再把这个缓冲区里的命令发给从节点，这样就保证了数据不丢失。
>
> 全量同步完成后，主从之间就会建立一个长连接。以后主节点每收到一个写命令，就会异步地发送给从节点，从节点跟着执行就好了。这期间他们还会互相发心跳包（Ping/Ack）来确认对方还活着。
>
> 如果发生网络抖动导致从节点掉线了一小会儿又连上了，如果每次这样都进行一次全量复制，太浪费了。所以redis后续在主节点内部增加一个环形的缓冲区叫 `repl_backlog_buffer`（积压缓冲区）。主节点会将写入命令存到这个缓冲区中，但是大小有限，待写入的命令超过 1m 后，会覆盖之前的数据，因为是环形写入。
>
> 如果从节点连上来，告诉主节点：“我刚才读到偏移量 1000 了”，主节点一看积压缓冲区里 1000 之后的数据还在，就会只把断线期间这部分数据补发给从节点，这就叫**增量复制**。
>
> 如果环形缓冲区被覆盖，那就只能全量复制了
>
> **注意：** redis允许链式结构，比如A挂了子节点B，B挂了子节点C，这样可以避免B和C都是A的子节点，让A在内部维护多个`Replication Buffer`

1、Redis的大key

> Redis里的big key指的是内存占用比较大的那些key，他们会导致
>
> * 在集群模式下如果big key都分配给了一个redis就会使得内存分布不均匀
> * 操作big key时由于redis时单线程执行命令，导致耗时严重，正常业务被卡死
> * 如果需要对big key进行网络传输，会严重消耗网络带宽，并且传输一个big key也非常容易出现超时的情况
> * 主从同步时，RDB以及AOF重写时复制big key都会很慢
>
> 解决big keys的办法：
>
> 1. 首先通过redis-cli --bigkeys -i 0.1（代表每扫描100个key休息0.1s）来查询bigkeys，但有一点需要注意，这个命令对于hash表/集合这种查的时候是以元素个数为标准，因为底层是 SCAN + STRLEN/HLEN/SCARD这些。但是元素多不一定占用内存大。所以还需要结合MEMORY USAGE key查看当前key实际占用的内存。
> 2. 找到big key后判断他是大key的原因，对于String是否是存储的JSON字段过于长，是否包含了不必要的业务字段？可以考虑仅存储必要的字段。对于Hash，set等来说，看看是不是所有元素都塞在了一个hash或set里。
>    * hash可以分桶，将大量数据分开存储，比如说假如一个Hash有100w个field，比如是HSET user:follow，可以拆成user:follow:0，user:follow:1...，user:follow:100
>    * List也可以，比如也可以按时间分队列
> 3. 删除big key一定不能使用del，因为一个big key的删除可以会阻塞好几秒，业务直接挂了。redis 4.0之前通过HSCAN， SSCAN，LTRIM等分批删除，redis4.0之后使用unlink，他会在主线程将big key从键空间剔除，后续占用的内存又其他线程在后台异步删除

2、Redis的过期策略

> redis的key过期后有两种删除策略，分别是惰性删除和主动删除。
>
> 惰性删除就是在查key时，如果判断key已经过期了就会删除，好处是不会消耗额外的资源，坏处就是如果一个key一直不用就一直不会被发现并删除
>
> 主动删除就是redis会定时开启一个任务，会主动扫描一部分key检查是否有过期的，通常控制在25ms。他只会删除一部分，而不是直接一次性扫描完。
>
> 除此之外Redis还有内存淘汰策略，redis提供了八种内存淘汰策略
>
> 1. no eviction 也就是不淘汰
> 2. allkeys-lru，针对所有key，删除最近最少使用的key
> 3. allkeys-lfu，针对所有key，删除最近使用频率最少的key
> 4. allkeys-random，针对所有key，随机删除
> 5. volatile-lru，针对设置了过期时间的key，删除最近最少使用的key
> 6. volatile-lfu，针对设置了过期时间的key，删除最近使用频率最少的key
> 7. volatile-random，针对设置了过期时间的key，随机删除
> 8. volatile-ttl，针对设置了过期时间的key，删除ttl最短的key

3、Redis是怎么找到key存储在哪个节点上

> 如果是redis cluster，首先会对查询的key执行CRC16算法，再将求得的数对16384进行取模确定该key属于哪个slot。如果当前连接的客户端就负责该slot那么直接返回，如果他不负责，则会抛出一个MOVED的错误，并告诉应该去连接的redis客户端，然后再去连接指定redis客户端查询key即可。（平时MOVED错误的处理一般都由SDK主动解决了）

6、MySQL的事务隔离级别，RR下怎么解决幻读的

> RR理论上只能解决快照读的幻读问题，对于当前读，如SELECT ... FOR UPDATE，SELECT ... LOCK IN SHARE MODE，UPDATE ..，DELETE ..，INSERT ...等他们是要加锁的读，必须读最新的版本，走的不是MVCC的机制而是锁机制。
>
> 对于当前读，必须依靠Next-Key Lock临间锁，他是Record Lock和Gap Lock的结合体，锁住的是当前读的数据以及他相邻记录之间的间隙。避免读的数据被修改以及在读的数据中插入新数据。

7、MySQL三层B+树能存储多少数据

> 首先innodb默认数据页大小是16kb，每个节点都有主键索引和指针，假设主键索引是bigInt占8字节，指针假设折中，占6字节。那么一个非叶子结点就能存16 * 1024 /（8+6）=1170个节点。
> 假设每条记录1kb，一个叶子结点存16条记录。
> 三层结构就是1170 * 1170 * 16 = 2190w条。差不多就是2000w-3000w左右
>
> 如果用uuid代替bigint，由于uuid是36字节，导致非叶子结点能存的节点大大减少，相同的数据，树更高，而排序树的查询性能与树高正相关。所以 uuid 做主键不光是占空间，还会让索引树变高，查询性能也受影响。
>
> **二级索引的三层b+树呢？**：二级索引的b+树叶子结点存的不是完整数据而是主键值，假设主键索引是big int，二级索引也是big int也就是都是8字节。那么一个叶子结点就能存16 * 1024 / (8 + 8) = 1000条。那么总的记录就是1170 * 1170 * 1000 = 13.5亿条

8、索引为什么需要最左前缀

> MySQL 索引的**最左前缀匹配原则**是指使用联合索引时，查询条件必须从索引的最左列开始匹配。联合索引包含多个列，查询时必须先有第一个列的条件，然后是第二个列，以此类推。
>
> 这是因为联合索引在 B+ 树中的排列方式是"从左到右"的顺序。比如联合索引 `(first_name, last_name, age)` 会先按 first_name 排序，first_name 相同再按 last_name 排序，last_name 相同再按 age 排序。
>
> MySQL 查找时会优先用 first_name 作为匹配依据，然后依次用 last_name 和 age。跳过最左侧字段，后面的列在 B+ 树中是无序的，压根没法利用索引快速定位。

11、Spring中Bean的初始化过程

> 1. 实例化，new出这个bean的对象
> 2. 属性注入，通过setter注入或者是构造器注入或者是字段注入，为bean对象填充属性
> 3. 如果实现类aware相关接口，那么允许在初始化之前通过实现的aware相关接口获取bean的名字，工厂，类加载器等等
> 4. 初始化前有个前置的BeanPostProcessor，也就是postProcessorBeforeInitialization，比如@PostConstruct的扫描就在这
> 5. 正式执行初始化方法，比如@PostConstruct修饰的方法，实现了InitializingBean的afterPropertiesSet方法，或者是自定义的init-method方法
> 6. 初始化完成后，还有后置的BeanPostProcessor，也就是postProcessorAfterInitialization，AOP的代理就是在这做的
> 7. 然后bean就会被放进一级缓存，就能使用了

1. 本地缓存怎么做的?数据要更新怎么办?

   > 1. 可以直接用Map，但是Map无法设置过期时间，无法主动淘汰，内存容易爆
   > 2. 更主流的选择是使用caffeine，他能够配置最大可使用的内存，避免内存在线上爆了。可以配置expireAfter属性，指定过期策略。比如expireAfterCreate，在创建后多久时间过期；expireAfterAccess，多久没访问后过期等等。性能非常好。也能设置主动过期的TTL。同样也支持直接删除
   >
   > 数据更新的话：
   >
   > 1. 更新db数据时，同时删除redis以及caffeine的数据，不做回写。因为缓存只负责“加速读取”，不做数据源
   > 2. 可以将删除操作异步化，比如使用redis stream或者是消息中间件MQ，更新数据库时发送一条消息来删除缓存。
   > 3. 等下次查询对应数据时，缓存未命中则利用分布式锁+双重检测，最大可能降低加锁的代价。提升性能，保证数据最终一致性。

   > 1. 客户端一直被阻塞等待，最终超时，导致用户体验不好
   > 2. http连接一直被占用，线程资源一直被持有。后续的请求只能排队
   >
   > 这些资源就有：
   >
   > 1. 线程资源，以Tomcat为例，一个请求就占用一个线程，那请求被阻塞线程并不会被释放，就导致一直占用线程资源，无法处理其他请求
   > 2. 数据库连接，如果慢操作里有数据库的相关操作，可能导致数据库连接一直无法释放，进而影响其他的查询接口
   > 3. cpu资源，如果某个方法执行慢是因为方法本身有许多复杂的计算逻辑，那就会消耗更多cpu的资源，导致整个机器变慢
   > 4. 内存资源，长时间执行一个方法，可能导致其内部使用到的对象一直无法被GC释放，导致内存一直被占用。可能会导致GC频繁，出发STW，导致业务更慢。甚至OOM

9. 继续说一下怎么处理上面这种情况?

   > 1. 做好对应的降低，熔断策略。
   > 2. 日志记录，资源监控等逻辑，异步处理，不要同步执行
   > 3. 使用缓存来缓解数据库的查询压力，提升查询性能

11. Mysql常用的存储引擎

    > * Innodb特点：支持事务ACID，支持行锁，支持崩溃恢复，支持外键，支持聚簇索引（索引与数据一起）
    > * MyISAM特点：
    >   * 非聚簇索引，索引与数据分离，因此主键索引和二级索引的叶子结点存的都是**数据行的物理地址**，所以不存在回表的说法。（innodb是聚簇索引，因此innodb范围查询和顺序查询等场景会更有优势，缺点就是插入时如果主键无序容易页分裂，单纯按二级索引查时需要回表。而MyISAM采用的非聚簇索引，单纯的读性能很高，无论是主键还是二级索引都是一次直接定位。缺点就是每次都是随机IO，范围和顺序查询性能不如innodb）
    >   * 只有表锁，不支持行锁。（读共享，写排他）一个写操作整表阻塞。而innodb只锁一行
    >   * 没有undoLog，redoLog，bufferPool。不支持事务ACID，不支持崩溃恢复
    >   * count(*)极快，O(1)，而innodb需要扫索引O(n)（因为MyISAM在表的元数据里维护了一个row_count表级行数计数器，但是仅限count(1和星)，count字段还是只有扫。而innodb因为有事务和mvcc，不同事务看到的行数可能不一样，需要按照当前事务的readview去统计。并且还有回滚和并发修改，想维护一个全局可靠的变量还是太复杂了，没必要。）
    >   * 支持**全文检索**（利用倒排索引的思想），但innodb5.6版本后也支持了
    >   * 空间占用小，无undolog。
    >   * 因为没有mvcc，没有事务，没有redolog，undolog等复杂机制。所以读性能很好。
    >   * 多用于做日志归档，只读分析等

12. Innodb 这种存储引擎它是怎么来帮助加速读写的?

    > 1. 使用buffer Pool将数据页缓存到内存中，将磁盘随机IO转换成内存读写，大幅提升性能。同时结合预读机制，当查询某个数据页时将相邻的数据页一并缓存，提升缓存命中率。并且结合redolog 使写操作的刷盘操作异步化。
    > 2. 使用聚簇索引，使用索引进行精准查询以及范围查询时性能更好，且不需要回表。同时连续的主键还能有效防止页分裂的发生，提升磁盘利用率
    > 3. 使用索引作为条件进行修改时使用行锁，锁的粒度更细，多线程并发写时锁竞争更小，性能更好
    > 4. 结合mvcc使得快照读不加锁
    > 5. 通过组提交group commit一次性同时提交多个事务，减少fsync的次数（应用程序执行write写数据时，只是将数据写到了内核的缓冲区page cache中，**page cache**是操作系统为**文件读写**准备的内存缓冲区。然后由操作系统根据当前线程数，内存压力，page cache的使用情况等决定何时将数据刷新到磁盘。当然应用也能主动发起fsync将page cache主动刷新到磁盘。）

14. Java 线程那个生命周期是什么样的?它各状态之间是怎么流转的?

    > 1. **NEW**：此时线程刚被new出来，还没调用start方法
    > 2. **RUNNABLE**：调用 start 后进入这个状态，注意这里包含了操作系统层面的就绪和运行两个状态，只要没被阻塞就算 RUNNABLE
    > 3. **BLOCKED**：线程未拿到synchronized锁，被挡在同步代码块外面阻塞住
    > 4. **WAITING**：调用来Object.wait、Thread.join 或 LockSupport.park，进入**无限期等待**，必须被别的线程显式唤醒
    > 5. **TIMED_WAITING**：和 WAITING 类似，但是带超时时间，是**限时等待**，比如 Thread.sleep、Object.wait(timeout)、Thread.join(timeout)
    > 6. **TERMINATED**：run 方法执行完毕或者抛了未捕获异常，线程挂了
    >
    > NEW 状态调用 start 方法进入 RUNNABLE。RUNNABLE 状态如果抢 synchronized 锁失败进入 BLOCKED，拿到锁回到 RUNNABLE。RUNNABLE 状态调用 wait/join/park 进入 WAITING，被 notify/unpark 唤醒回到 RUNNABLE。RUNNABLE 状态调用 sleep/wait(timeout)/join(timeout) 进入 TIMED_WAITING，超时或被唤醒回到 RUNNABLE。RUNNABLE 状态 run 方法结束或异常进入 TERMINATED。

15. 什么情况下会出现死锁?死锁会带来什么影响?

    > **死锁（Deadlock）指的是两个或多个线程在执行过程中，因为争夺资源而造成一种互相等待的现象**，如果没有外力干预，它们都无法继续执行下去。
    >
    > 通俗说就是：A 等 B 手里的资源，B 又等 A 手里的资源，结果两人谁也不让步，程序就卡住了。
    >
    > 经典操作系统理论指出，死锁必须同时满足以下四个条件：
    >
    > 1. **互斥条件（Mutual Exclusion）**
    >
    >    - 资源不可共享，某一时刻只能被一个线程占用。
    >    - 例：文件、数据库连接、锁对象等。
    >
    > 2. **占有并等待（Hold and Wait）**
    >
    >    - 线程已经占有了至少一个资源，同时又申请新的资源而被阻塞。
    >
    > 3. **不可剥夺（No Preemption）**
    >
    >    - 已经分配给线程的资源不能被强制剥夺，只能线程自己释放。
    >
    > 4. **循环等待（Circular Wait）**
    >
    >    - 存在一种线程循环等待资源的环形链：
    >
    >      ```
    >      T1 等待 T2 持有的资源 R2
    >      T2 等待 T3 持有的资源 R3
    >      ...
    >      Tn 等待 T1 持有的资源 R1
    >      ```
    >
    > **死锁带来的影响**
    >
    > 1. **程序卡死**
    >    - 死锁线程无法继续执行，整个业务可能阻塞。
    > 2. **系统性能下降**
    >    - 资源被占用，CPU 或内存可能空转等待。
    > 3. **数据不一致 / 锁泄露**
    >    - 业务资源长期被占用，可能导致数据异常或服务不可用。
    > 4. **难以排查**
    >    - 死锁通常只在特定并发条件下出现，排查困难。
    >
    > **常用的死锁解决策略**
    >
    > 1. **破坏互斥**（可行性小，多用于资源共享）
    > 2. **破坏占有并等待**
    >    - 一次性申请所有资源，避免持有一部分资源后再等待。
    > 3. **破坏不可剥夺**
    >    - 如果请求不到资源就释放已占有的资源，重新尝试。
    > 4. **破坏循环等待**
    >    - 给资源编号，按顺序申请，避免环形等待。
    > 5. **超时 / watchdog 机制**
    >    - 线程等待超过一定时间，主动放弃资源。
