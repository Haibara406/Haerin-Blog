### 微服务多点部署其中一个宕机了怎么办？

正常情况下如果是多实例部署的话，其中一个实例宕机了一般不会影响整体服务。因为服务会通过服务发现 + 复杂均衡 + 健康检查来自动处理。核心思路就是：**自动摘除故障实例，让请求打到其他正常实例上**。

比如说现在订单服务部署了三个实例，项目中会将这些实例都注册到服务注册中心，比如nacos。

那客户端现在的访问流程就是 客户端 -> gateway -> 负载均衡 -> 3个订单服务实例

如果某一个实例宕机了，nacos会通过心跳机制或者是健康检查，监测到某个实例的心跳消失，比如nacos默认：30s 无心跳 → 标记不健康 / 90s 无心跳 → 删除实例

注册中心将实例摘除后负载均衡就会自动跳过宕机的实例，用户基本感知不到。

而对于已经打到的宕机实例的请求最好的方式就是：

* 通过重试机制，前几次请求都打到宕机的实例上，后续的请求会因为宕机实例的摘除会通过负载均衡打在健康的实例上
* 如果失败率很高，可以通过降级或者熔断的机制，返回友好错误，避免雪崩

### 如果一个微服务的所有实例都宕机了怎么办？

这就属于系统级别的故障，一般应对的措施就是：

* 如果所有实例都宕机了，应该通过降级策略快速失败，返回默认降级结果，避免客户端请求长时间阻塞
* 通过熔断机制，将宕机的服务都熔断出去，避免宕机后仍有大量请求打进服务器，造成进一步故障
* 使用缓存数据兜底，查询请求来时通过caffeine或者是redis进行缓存数据兜底
* 使用MQ将写操作异步化，或者将请求暂存起来，等实例恢复后，再逐一消费
* 使用备用的服务做兜底，比如AI自动化推荐服务的实例宕机了，可以用默认热门推荐服务顶替，虽然推荐变得不精准，但功能仍可使用。
* 很多公司其实都会进行多机房多地域容灾，比如位于成都的机房挂了，可以通过DNS或者是网关将流量切换到上海的机房

### 要是mq占据大量CPU该怎么排查？MySQL占据大量CPU该怎么排查？

首先，无论是哪个进程出现CPU飙升，通常都意味着系统正在进行**高频计算**、**大量 IO 等待**或是**陷入了某种死循环**。

#### MQ

对于mq而言排查的流程为：

1. 通过linux的`top` 或者 `top -Hp <mq-pid>` 命令查看是不是MQ的进程cpu占用高
2. 如果找到了cpu占用高的mq进程，再定位到具体的线程。通过先通过top查看到对应进程的PID，然后通过printf "%x\n" PID 拿到十六进制的PID，这一步主要是方便用Java写的MQ，比如RocketMQ和Kafka
3. 如果是Java写的MQ比如RocketMQ和Kafka，可以通过`jstack <pid> | grep 3039 -A 30`查看具体进程里的线程堆栈信息。通过分析堆栈信息，可能会发现原因是：消费者处理逻辑慢，消费者代码有问题（比如有复杂计算，调用了慢接口，循环层级太多或者有死循环），或者是消息很多导致消息堆积进而消费者压力很大。而且像Kafka，Rocket持久化时需要刷盘，如果磁盘慢，cpu可能也会受影响。

> 因此对于MQ主要分析：消息堆积，消费速率，重试次数，IO

#### Mysql

对于Mysql而言，分析排查的流程为：

1. 首先就是通过top命令查看是哪个Mysql进程cpu占用高
2. 然后登录到对应的mysql里，通过show full processlist查看当前所有连接到数据库的“线程”信息。
   * 主要关注time持续时间， 单位是**秒**。表示当前这个状态已经持续了多久。数值越大越可疑。
   * state详细状态，它描述了 SQL 执行的具体阶段（如 `Sending data` 或 `Sorting`）。
   * Info，具体的SQL语句
   * Command，表示线程正在执行的操作。常见值：`Sleep`（空闲）、`Query`（正在查询）、`Connect`（正在连接）
3. 同样也可以查询慢查询日志, `show variables like 'slow_query_log';` 然后对里面的sql执行explain，看看是否使用了索引，索引的区分度是否高，where过滤的数据是否多，以及对应的extra额外信息等。
4. 查看innodb的状态，比如 `show engine innodb status;` 看是否有激烈的锁竞争，死锁等

最终基本就是通过优化索引或改写 SQL 解决问题。

### 如何设计一个点赞系统

点赞系统虽然听起来简单，无非就是点赞+1，取消-1，然后展示总点赞数。但实际上他的核心在于点赞系统的核心挑战在于**高并发写入**和**数据一致性**的平衡。

比如像抖音、微博、b站这种日活过亿的 App，热门内容一秒钟能涌进几万个点赞请求，直接怼数据库铁定崩。

因此核心思路就是：**缓存抗流量、异步削峰、批量落库**。

1. 请求全部打到redis中，然后使用redis的set集合来操作数据。SADD添加点赞用户，SREM删除取消点赞的用户，SISMEMBER快速判断用户是否点赞，SCARD快速获取总点赞数
2. Redis开启RDB/AOF做持久化，但RDB和AOF仍然会导致短时间内的数据丢失，因此需要使用消息队列异步发送消息，消费者负责将Redis中的数据批量同步到Mysql中，做数据持久化

核心的数据库表其实就两张，一个点赞数表，一个点赞记录表

```sql
-- 点赞记录表，记录谁给哪个内容点了赞
CREATE TABLE like_record (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    item_id BIGINT NOT NULL,
    liked_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY unique_user_item (user_id, item_id)
) ENGINE=InnoDB;

-- 点赞计数表，存每个内容的总点赞数
CREATE TABLE like_count (
    item_id BIGINT PRIMARY KEY,
    like_count BIGINT DEFAULT 0
) ENGINE=InnoDB;
```

那么既然用了缓存，就必然会涉及数据一致性问题。点赞场景对一致性要求没那么高，用户晚几秒看到最新点赞数完全可以接受，所以一般采用**最终一致性**方案。

具体做法是：Redis 里的点赞数是实时的，MySQL 里的数据通过定时任务或消息队列异步同步。

如果 Redis 挂了，从 MySQL 加载数据重建缓存。

有个细节要注意：取消点赞的时候不能简单用 DECR，因为如果消息丢了或者重复消费，点赞数可能变成负数。

稳妥的做法是用 Redis 的 SET 数据结构存点赞用户集合，点赞就 SADD，取消就 SREM，点赞数用 SCARD 获取。

然后可以设一个定时任务每天凌晨运行任务，对比 Redis 与 MySQL 的点赞数，以 MySQL 为准覆盖 Redis，做数据校准。

当然每次展示点赞数都执行一次SCARD或者ZCARD也比较消耗性能，可以将数据存储在redis的hash结构中，O(1)

#### 分库分表 / 分片

当数据量上去之后单表肯定扛不住，得分库分表。

点赞记录表按 item_id 取模分表比较合适，这样查某个内容的所有点赞记录只需要查一张表。如果按 user_id 分，查某个内容的点赞数就得扫所有分表，效率太低。当然如果需要支持查看用户点赞的内容可以在item_id的基础上注入user_id的基因

点赞计数表数据量相对小，可以不分表，或者按 item_id 范围分表。

而在redis里面，对于热度高的视频或者是一些大V，一个视频会有上百万个点赞，如果所有数据都放在一个set里，肯定会导致big key，进而增加cpu消耗，以及造成redis卡顿等问题。

因此可以考虑通过hash tag分散一下set的数据到多个set里，可以有效缓解big key的相关问题。比如可以将一个set集合分成64个小的set集合，用户点赞时根据 `user_id` 进行哈希取模，分散到不同的桶中。

然后对于一些时间过了很久的视频，热度下降后，将redis中的数据同步到mysql中后，就可以删除redis中的数据，释放对于内存。

还有就是可以再在redis前再加一层本地缓存，热门视频的点赞数直接缓存在应用服务器的内存里，用 Guava Cache 或 Caffeine。本地缓存的过期时间设短一点，比如 1 秒，这样既能抗住瞬时流量，数据也不会太旧。

#### 防刷

点赞系统必须防刷，不然竞争对手雇一堆水军把你服务器刷挂。常见手段：

1）限流：单用户每秒最多点赞 5 次，用 Redis 的滑动窗口或令牌桶实现

2）频率检测：同一个 IP 短时间内给同一个内容点赞超过阈值，直接拉黑

3）验证码：异常行为触发验证码校验

4）延迟生效：点赞数不实时展示，延迟几分钟，给风控系统留出识别时间

#### 朋友圈点赞场景

对于朋友圈点赞的场景，有一个核心功能：朋友圈需要展示点赞的用户头像列表信息。这本质上其实就是使用ZSET里的score字段记录点赞时间，然后按照点赞的先后顺序展示用户头像/用户名等

具体的设计就是：

1. zset 的 key 设为朋友圈动态 ID
2. score 设为点赞时间戳
3. value 设为点赞用户 ID

**点赞操作**：用 ZADD 命令把用户 ID 和当前时间戳写入 zset，时间复杂度 O(logN)。zset 天然去重，同一个用户重复点赞不会产生多条记录。

**取消点赞**：用 ZREM 命令把用户 ID 从 zset 中删掉，时间复杂度 O(logN)。

**获取点赞列表**：用 ZRANGE key 0 -1 拿到所有点赞用户，按时间正序排列。如果想按时间倒序展示最新点赞的人，用 ZREVRANGE 就行。时间复杂度 O(logN+M)，M 是返回的元素数。

同样的也可以根据userId基于hash tag对zset进行分片存储，避免big key。

#### 展示点赞列表的时候，怎么只显示共同好友？

最简单的就是讲好友列表也加入redis的set集合，然后让这两个集合通过SINTER取交集
